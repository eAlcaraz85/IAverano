*  Tutorial — Entornos virtuales en Python (Windows, Linux, macOS)




Aprender a crear y usar *entornos virtuales* en Python cuando tienes *varias versiones* de Python instaladas (3.9, 3.10, 3.11, 3.12, 3.13…) en:

- *Windows*
- *Linux* (Debian/Ubuntu, Fedora, Arch…)
- *macOS* (Intel y Apple Silicon)

Al terminar sabrás:

1. Qué es un entorno virtual y por qué no instalar todo con =pip= global.
2. Cómo ver qué versiones de Python tienes en cada sistema.
3. Cómo crear un =venv= con una versión *concreta*.
4. Cómo gestionar muchas versiones con *pyenv* (recomendado).
5. Cómo activar, desactivar, exportar dependencias y borrar entornos.
6. Cómo resolver errores típicos.

** Conceptos básicos

*** ¿Qué es un entorno virtual?
Un *entorno virtual* es una carpeta aislada que contiene:

- Un intérprete de Python (o un enlace a uno concreto).
- Su propio =pip=.
- Sus propios paquetes en =site-packages=.

Cada proyecto puede tener versiones distintas de Django, NumPy, etc., sin pelearse.

*** Entorno virtual ≠ versión de Python
| Idea | Qué es |
|------+--------|
| *Versión de Python* | 3.10, 3.12… el lenguaje en sí |
| *Entorno virtual* | Carpeta aislada de paquetes *para un proyecto* |

Puedes tener Python 3.12 instalado y crear tres entornos 3.12 con librerías diferentes.

*** Herramientas que verás en este tutorial
| Herramienta | Viene con Python | Varias versiones de Python | Uso típico |
|-------------+------------------+----------------------------+------------|
| =venv= | Sí (3.3+) | Solo si ya tienes ese binario | Proyectos simples, estándar |
| =virtualenv= | No (=pip install=) | Igual que =venv= | Compatibilidad extra |
| *pyenv* | No | *Sí*, instala y cambia versiones | Desarrollo con 3.9–3.13 |
| *conda* / Miniconda | No | Sí (canales propios) | Ciencia de datos, CUDA |
| *uv* | No | Sí, muy rápido | Proyectos modernos (mención) |

*Recomendación general:*
- *Principiante / curso:* =python3.12 -m venv .venv= (o la versión que pida tu materia).
- *Varias versiones en la misma máquina:* *pyenv* + =pyenv virtualenv=.

** Método 1 — =venv= (incluido en Python, el más estándar)

*** Crear entorno con una versión específica

La regla de oro:

*** Activar el entorno

| Sistema | Comando activar | Desactivar |
|---------+-----------------+------------|
| Linux / macOS | =source .venv/bin/activate= | =deactivate= |
| Windows CMD | =.venv\Scripts\activate.bat= | =deactivate= |
| Windows PowerShell | =.venv\Scripts\Activate.ps1= | =deactivate= |

*PowerShell* puede bloquear scripts. Si falla:

#+begin_src powershell :exports code

#+end_src

*** Comprobar que usas el Python correcto

Tras activar:

#+begin_src shell :exports code

#+end_src

En Windows:

#+begin_src powershell :exports code
Get-Command python
python --version
pip --version
#+end_src

Debe apuntar a =.venv= (Linux/Mac) o =.venv\Scripts\python.exe= (Windows).

*** Instalar paquetes y guardar dependencias

#+begin_src shell :exports code

#+end_src

En otra máquina o compañero:

Solo elimina la carpeta (con el entorno *desactivado*):

#+end_src

** Método 2 — *pyenv* (varias versiones de Python en una máquina)

*** ¿Cuándo usar pyenv?
- Necesitas 3.10 para un proyecto viejo y 3.12 para uno nuevo.
- No quieres mezclar el Python del sistema (Linux/macOS) con tus proyectos.
- Quieres un comando global: =pyenv local 3.11.9= en una carpeta.

*** Instalación

**** Linux (Debian/Ubuntu — dependencias + instalador oficial)

#+begin_src shell :exports code
sudo apt update
sudo apt install -y make build-essential libssl-dev zlib1g-dev \
  libbz2-dev libreadline-dev libsqlite3-dev curl git \
  libncursesw5-dev xz-utils tk-dev libxml2-dev libxmlsec1-dev libffi-dev liblzma-dev

curl https://pyenv.run | bash
# Añade a ~/.bashrc o ~/.zshrc:
# export PYENV_ROOT="$HOME/.pyenv"
# export PATH="$PYENV_ROOT/bin:$PATH"
# eval "$(pyenv init -)"
#+end_src

Reinicia la terminal y:

#+begin_src shell :exports code
pyenv --version
#+end_src

**** macOS (Homebrew)

#+begin_src shell :exports code
brew update
brew install pyenv
# Añade a ~/.zshrc:
# export PYENV_ROOT="$HOME/.pyenv"
# export PATH="$PYENV_ROOT/bin:$PATH"
# eval "$(pyenv init -)"
#+end_src

**** Windows — *pyenv-win*

En Windows el proyecto equivalente es [[https://github.com/pyenv-win/pyenv-win][pyenv-win]] (no es el mismo binario que Linux, pero la idea es igual).

winget install pyenv-win.pyenv-win

# O clonar en %USERPROFILE%\.pyenv\pyenv-win
# y añadir a PATH las carpetas bin y shims
#+end_src

Consulta la wiki de pyenv-win para variables =PYENV=, =PYENV_ROOT=, =PYENV_HOME= en *Variables de entorno* de Windows.

#+end_src

** Método 3 — =virtualenv= (paquete pip)

Similar a =venv=, útil si necesitas opciones extra o Python muy antiguo.

#+begin_src shell :exports code
python3.12 -m pip install --user virtualenv
python3.12 -m user virtualenv .venv
# o:
virtualenv -p python3.11 .venv311
source .venv311/bin/activate
#+end_src

En la práctica moderna, =python -m venv= suele bastar.

** Método 4 — Conda / Miniconda (opcional, ciencia de datos)

*** Idea
Conda crea entornos con Python *y* librerías binarias (NumPy, CUDA) desde canales propios.

#+begin_src shell :exports code
conda create -n mi_env python=3.11
conda activate mi_env
conda install pandas numpy
conda deactivate
#+end_src

*** Conda vs venv

| Aspecto | venv / pyenv | conda |
|---------+--------------+-------|
| Peso | Ligero | Más pesado |
| Paquetes no-Python | No | Sí (R, C libs) |
| Curso general IA/Python | venv suele bastar | Si el curso pide Anaconda |

** Flujo de trabajo recomendado por escenario

*** Escenario A — Un solo proyecto, una versión fija (curso)

#+begin_example
1. Instalar Python 3.12 desde python.org / apt / brew
2. cd proyecto
3. python3.12 -m venv .venv
4. activar → pip install -r requirements.txt
5. Añadir .venv/ al .gitignore
#+end_example

*** Escenario B — Tres proyectos, tres versiones

#+begin_example
1. Instalar pyenv
2. pyenv install 3.10.13 3.11.9 3.12.4
3. En cada repo: pyenv local X.Y.Z && python -m venv .venv
#+end_example

*** Escenario C — Windows con lanzador py

#+begin_example
1. py -0p   (ver versiones)
2. py -3.12 -m venv .venv
3. .venv\Scripts\Activate.ps1
4. pip install ...
#+end_example

** Buenas prácticas

1. *Un entorno por proyecto* (carpeta =.venv= o =venv= dentro del repo).
2. *No subas* =.venv= a Git; sí sube =requirements.txt= o =pyproject.toml=.
3. *Nombra* la versión en el README: “Requiere Python ≥ 3.11”.
4. En Linux, evita =sudo pip install= en el Python del sistema (rompe paquetes del SO).
5. En macOS con Apple Silicon, prefiere builds *arm64* (Homebrew en =/opt/homebrew=).
6. Usa =python -m pip install= en lugar de =pip= suelto (asegura el pip del entorno activo).

** Problemas frecuentes y soluciones

| Síntoma | Causa probable | Solución |
|---------+----------------+----------|
| =python: command not found= | Python no en PATH | Instalar o usar ruta completa / pyenv |
| =No module named venv= | Paquete =python3-venv= falta (Debian) | =sudo apt install python3.12-venv= |
| Tras activar, =python= sigue siendo el del sistema | No activaste bien o orden PATH | =source .venv/bin/activate=; =which python= |
| PowerShell no activa | ExecutionPolicy | =RemoteSigned= para CurrentUser |
| =pip install= instala fuera del venv | Entorno no activo | Activar; usar =python -m pip= |
| pyenv: version not installed | Falta =pyenv install= | =pyenv install 3.12.4= |
| Mezcla 3.10 y 3.12 en mismo venv | Creaste venv con binario equivocado | Borrar .venv y recrear con binario correcto |

* Introducción a la Inteligencia Artificial

Este documento desarrolla el temario de la unidad introductoria a la *Inteligencia Artificial (IA)*: definición, historia, bases cognitivas, lógica, filosofía del conocimiento, modelos cognitivos y de agente, y el papel de la *heurística* con algoritmos de búsqueda.

*Objetivos generales:*
- Comprender qué se entiende por IA y cómo ha evolucionado históricamente.
- Relacionar la IA con psicología, lógica y filosofía del conocimiento.
- Describir el *modelo del agente inteligente* y sus extensiones (multiagente, ubicuo).
- Aplicar algoritmos de búsqueda informada y local como base de la resolución de problemas.

** 1.1 Introducción a la Inteligencia Artificial

*** ¿Qué es la Inteligencia Artificial?
La *Inteligencia Artificial* es el campo de la informática y las ciencias cognitivas que estudia cómo construir sistemas capaces de realizar tareas que, si las hiciera un humano, requerirían *inteligencia*: percibir, razonar, aprender, planificar, comunicarse y actuar en entornos complejos.

Definiciones frecuentes en cursos introductorios:
- *Sistemas que piensan como humanos* (enfoque cognitivo / simulación).
- *Sistemas que actúan racionalmente* (enfoque del *agente racional*; muy usado en IA moderna).
- *Sistemas que piensan racionalmente* (enfoque lógico: leyes del pensamiento).

*** Problemas típicos de la IA
| Área              | Ejemplos                                              |
|-------------------+-------------------------------------------------------|
| Percepción        | Visión por computadora, reconocimiento de voz         |
| Razonamiento      | Demostración de teoremas, diagnóstico médico            |
| Aprendizaje       | Clasificación, predicción, *deep learning*              |
| Lenguaje natural  | Traducción, chatbots, resumen de textos               |
| Planificación     | Robótica, logística, juegos (ajedrez, Go)             |
| Representación    | Ontologías, grafos de conocimiento, bases de reglas     |

*** IA débil vs IA fuerte (distinción conceptual)
- *IA débil (narrow AI):* sistemas especializados en una tarea (recomendador, detector de spam, asistente de voz). Es lo que existe hoy en producción.
- *IA fuerte (AGI):* hipotética inteligencia general al nivel humano o superior en *cualquier* dominio. Sigue siendo objeto de debate y especulación.

*** Relación con otras disciplinas
- *Psicología cognitiva:* modelos de memoria, atención, aprendizaje.
- *Lógica y matemáticas:* formalización del razonamiento.
- *Filosofía:* naturaleza del conocimiento, mente y consciencia.
- *Neurociencia:* inspiración biológica (redes neuronales).
- *Ingeniería:* sistemas que funcionan en el mundo real (robots, sensores, redes).

*** Idea clave del curso
La IA no es un solo método: es un *conjunto de técnicas* (búsqueda, lógica, probabilidad, aprendizaje automático, etc.) unidas por la pregunta: *¿cómo representar el conocimiento y elegir acciones útiles en un entorno?*

** 1.2 Historia de la Inteligencia Artificial

*** Orígenes (décadas de 1940–1950)
- *Alan Turing* (1950): artículo “Computing Machinery and Intelligence”; propone el *Test de Turing* como criterio operativo de comportamiento inteligente.
- *McCulloch y Pitts* (1943): modelo neuronal binario (antecedente de redes neuronales).
- Conferencia de *Dartmouth* (1956): McCarthy, Minsky, Shannon, Rochester y otros acuñan el término *“Artificial Intelligence”*. Se inicia el optimismo de que la inteligencia general sería alcanzable en pocos años.

*** Era del optimismo (1956–1974)
- Programas pioneros: *Logic Theorist*, *General Problem Solver* (GPS), primeros juegos (damas, ajedrez simplificado).
- Lenguaje *LISP* (McCarthy, 1958) como herramienta simbólica.
- Expectativas altas; financiamiento abundante.

*** Primer “invierno” de la IA (1974–1980)
- Limitaciones computacionales y teóricas (combinatoria explosiva, conocimiento difícil de codificar).
- Informe *Lighthill* (Reino Unido, 1973): crítica severa → recortes de fondos.

*** Sistemas expertos y boom comercial (1980–1987)
- *Sistemas basados en reglas* (IF–THEN): MYCIN (diagnóstico), DENDRAL (química).
- Japón: *Fifth Generation*; EE.UU.: proyectos DARPA.
- Éxito en dominios acotados con expertos humanos disponibles.

*** Segundo invierno (1987–1993)
- Caída del hardware LISP; sistemas expertos caros de mantener y frágiles fuera de su nicho.
- Necesidad de *aprendizaje* y *datos* frente a solo reglas manuales.

*** Resurgimiento: aprendizaje automático y datos (1990–2010)
- *Aprendizaje estadístico*: SVM, árboles, *boosting*.
- Éxitos prácticos: filtrado de spam, reconocimiento de escritura, recomendación.
- *Deep Blue* vence a Kasparov (1997) con búsqueda + heurísticas + hardware.

*** Era del *deep learning* (2010–presente)
- GPUs, grandes conjuntos de datos (ImageNet), arquitecturas profundas (CNN, RNN, Transformers).
- Hitos: AlexNet (2012), AlphaGo (2016), GPT y modelos de lenguaje a gran escala (2020+).
- IA generativa, visión, voz, asistentes conversacionales.

*** Línea de tiempo resumida

#+begin_example
1943  Modelo McCulloch-Pitts
1950  Test de Turing
1956  Dartmouth → nace el término "IA"
1960s GPS, ELIZA, optimismo
1974  Primer invierno (Lighthill)
1980s Sistemas expertos
1987  Segundo invierno
1997  Deep Blue vs Kasparov
2012  AlexNet / revolución deep learning
2016  AlphaGo
2020+ LLMs, IA generativa
#+end_example

** 1.3 Habilidades cognoscitivas y teorías de la inteligencia

*** Habilidades cognoscitivas (psicología)
Desde la psicología cognitiva, la “inteligencia” en humanos se manifiesta en procesos interrelacionados:

| Proceso        | Descripción breve                                      |
|----------------+--------------------------------------------------------|
| Percepción     | Interpretar estímulos sensoriales                      |
| Atención       | Seleccionar información relevante                      |
| Memoria        | Almacenar y recuperar (corto/largo plazo)              |
| Lenguaje       | Comprender y producir símbolos lingüísticos            |
| Razonamiento   | Inferir conclusiones, resolver problemas                 |
| Aprendizaje    | Modificar conducta o conocimiento con la experiencia     |
| Metacognición  | “Pensar sobre el propio pensamiento”                   |

La IA clásica intentó *modelar* o *replicar* estos procesos; la IA moderna a menudo los *aproxima* con datos y optimización sin pretender fidelidad biológica.

*** Teoría conductista (conductismo)
- *Enfoque:* la mente no se observa directamente; se estudia el *comportamiento* ante estímulos y reforzadores.
- *Skinner, Watson:* aprendizaje por condicionamiento y refuerzo.
- *Relevancia en IA:* *aprendizaje por refuerzo* (RL): agente, estados, acciones, recompensas (Q-learning, policy gradient). El “conductismo” inspira el marco matemático, no la negación de representaciones internas (los agentes RL sí mantienen valores o políticas).

*** Teoría de la inteligencia general (factor *g*)
- *Spearman:* un factor común subyace a varias pruebas cognitivas.
- En IA: analogía limitada; los sistemas suelen ser *especialistas* (alto rendimiento en una tarea, bajo en otras).

*** Inteligencias múltiples (Howard Gardner)
Gardner propone varias inteligencias relativamente independientes:

| Inteligencia        | Ejemplo humano              | Paralelo en IA (aproximado)        |
|---------------------+-----------------------------+------------------------------------|
| Lingüística         | Leer, escribir, debatir     | NLP, LLMs                          |
| Lógico-matemática   | Demostrar, programar        | Lógica, planificación, ML simbólico|
| Espacial            | Orientarse, dibujar         | Visión, SLAM, gráficos             |
| Musical             | Componer, reconocer tonos   | Generación/análisis musical        |
| Corporal-kinestésica| Deportes, manualidades      | Robótica, control motor            |
| Interpersonal       | Empatía, liderazgo          | Análisis de sentimientos (parcial) |
| Intrapersonal       | Autoconocimiento            | Poco modelado explícitamente       |
| Naturalista         | Clasificar especies         | Clasificación de imágenes          |

*Idea pedagógica:* no existe un solo “botón de inteligencia”; los sistemas actuales combinan módulos especializados.

*** Otras teorías (mención)
- *Piaget:* desarrollo por etapas (sensoriomotor → formal). Útil como metáfora de complejidad creciente en agentes que aprenden.
- *Vygotsky:* aprendizaje social y *zona de desarrollo próximo*. Paralelo: *aprendizaje con humano en el bucle* (RLHF, anotación humana).
- *Sternberg (triárquica):* analítica, creativa, práctica → diseño de sistemas que equilibren precisión, novedad y utilidad en contexto.

** 1.4 Razonamiento según la lógica

*** Lógica como fundamento del razonamiento automático
La lógica formal permite:
- Representar *proposiciones* y *relaciones*.
- Derivar *conclusiones válidas* a partir de *premisas*.
- Verificar si un argumento es *correcto* (independientemente de si las premisas son verdaderas en el mundo real).

*** Componentes básicos
- *Lenguaje formal:* símbolos, conectivos (∧, ∨, ¬, →), cuantificadores (∀, ∃).
- *Axiomas:* proposiciones *asumidas* sin demostración dentro del sistema (punto de partida).
- *Teoremas:* proposiciones *demostrables* a partir de axiomas y reglas de inferencia.
- *Demostración:* secuencia finita de pasos donde cada paso es un axioma, una hipótesis o se obtiene por una *regla de inferencia* (p. ej. *modus ponens*: de P y P→Q inferir Q).

*** Ejemplo mínimo (lógica proposicional)
Premisas:
1. Si llueve, la calle está mojada.  (Lluvia → Mojada)
2. Está lloviendo.                    (Lluvia)

Conclusión (por modus ponens): La calle está mojada. (Mojada)

En notación: {(Lluvia → Mojada), Lluvia} ⊢ Mojada

*** Sistemas formales relevantes en IA
| Sistema              | Uso en IA                                      |
|----------------------+------------------------------------------------|
| Lógica proposicional | Reglas simples, verificación finita            |
| Lógica de predicados | Relaciones, bases de conocimiento              |
| Lógica modal         | Conocimiento, creencia, obligación (agentes)   |
| Lógica difusa        | Razonamiento con grados de verdad              |

*** Límites importantes
- *Decidibilidad:* algunos problemas no tienen algoritmo que siempre termine con respuesta correcta.
- *Complejidad:* el espacio de búsqueda de demostraciones crece muy rápido → hace falta *heurística* (sección 1.8).
- *Mundo real:* la lógica clásica asume certeza; en la práctica hay incertidumbre (probabilidad, redes bayesianas).

** 1.5 Adquisición del conocimiento según la filosofía

*** ¿Qué es el conocimiento? (epistemología)
La *filosofía del conocimiento* pregunta:
- ¿Qué distingue *conocimiento* de mera *creencia*?
- ¿De dónde proviene? (experiencia vs razón)
- ¿Cómo sabemos que algo es *justificado*?

Tradición clásica: *conocimiento = creencia verdadera justificada* (definición discutida, pero pedagógicamente útil).

*** Empirismo vs racionalismo
| Corriente      | Idea principal                         | Paralelo en IA                          |
|----------------+----------------------------------------+-----------------------------------------|
| Empirismo      | Conocimiento desde la experiencia      | Aprendizaje automático, datos, inducción|
| Racionalismo   | Conocimiento desde la razón innata     | Axiomas, ontologías, reglas a priori    |

Los sistemas reales suelen combinar *conocimiento codificado* (reglas, ontologías) con *conocimiento aprendido* (modelos entrenados).

*** Fuentes de conocimiento en sistemas inteligentes
1. *Codificación manual:* expertos escriben reglas (sistemas expertos).
2. *Aprendizaje inductivo:* generalizar patrones desde ejemplos.
3. *Adquisición semi-automática:* extracción de texto, minería de datos.
4. *Interacción:* el usuario corrige o etiqueta (aprendizaje activo).

*** Problema de la adquisición de conocimiento (*knowledge acquisition bottleneck*)
- Los expertos no siempre saben articular lo que hacen.
- El mantenimiento de reglas es costoso.
- Motivación histórica del *machine learning*: aprender reglas o parámetros desde datos.

*** Representación del conocimiento (puente filosofía → IA)
- *Hechos:* “París es capital de Francia.”
- *Reglas:* “Si X es padre de Y, Y es hijo de X.”
- *Ontologías:* vocabulario formal de conceptos y relaciones.
- *Frames / redes semánticas:* estructuras con herencia y slots.

** 1.6 El modelo cognoscitivo

*** Definición
Un *modelo cognoscitivo* es una representación computacional de procesos mentales: cómo se percibe, se recuerda, se razona y se decide. Busca coherencia con evidencia psicológica o neurocientífica (a veces de forma simplificada).

*** Arquitecturas cognitivas clásicas
| Arquitectura | Idea central                                      |
|--------------+---------------------------------------------------|
| SOAR         | Memoria, aprendizaje por *chunking*, resolución de impasse |
| ACT-R        | Módulos (declarativo/procedimental), activación   |
| CLARION      | Conocimiento implícito vs explícito               |

*** Componentes típicos de un modelo cognoscitivo
- *Memoria de trabajo:* buffer limitado para información actual.
- *Memoria a largo plazo:* hechos y procedimientos.
- *Percepción y motor:* interfaces con el entorno.
- *Aprendizaje:* consolidación de experiencias en conocimiento reutilizable.

*** Modelo cognoscitivo vs modelo de agente
| Aspecto           | Modelo cognoscitivo           | Agente inteligente (IA clásica)   |
|-------------------+-------------------------------+-----------------------------------|
| Objetivo          | Simular mente humana          | Actuar bien en un entorno         |
| Evaluación        | Ajuste a datos psicológicos   | Rendimiento (tarea, utilidad)     |

Las redes neuronales profundas no suelen pretender ser modelos cognitivos fieles, pero incorporan ideas inspiradas en cerebro (capas, atención). Los *LLMs* debaten si emergen capacidades “cognitivas” sin modelo explícito de mente.

** 1.7 Modelo del agente inteligente

*** Definición de agente
Un *agente* es cualquier cosa que percibe su *entorno* mediante *sensores* y actúa sobre él mediante *actuadores*.

Un *agente inteligente* elige acciones que maximizan una medida de *rendimiento* según su *secuencia perceptiva* y su *conocimiento* incorporado.

*** Función agente
\[ f: P^* \rightarrow A \]
donde P es el conjunto de percepciones y A el de acciones.

En la práctica se implementa con *programa + arquitectura* (cuerpo físico o software).

*** Componentes de un agente racional
1. *Medida de rendimiento:* qué se considera éxito (seguridad, puntos, tiempo, costo).
2. *Conocimiento del entorno:* reglas, mapas, modelos probabilísticos.
3. *Secuencia perceptiva:* historial de lo observado.
4. *Acción:* la mejor según expectativa de rendimiento futuro.

*** Tipos de agentes (Jersey / Russell & Norvig)
| Tipo                    | Comportamiento                                      |
|-------------------------+-----------------------------------------------------|
| Reflejo simple          | Acción = función de percepción actual               |
| Reflejo basado en modelo| Mantiene estado interno (modelo del mundo)          |
| Basado en objetivos     | Planifica para alcanzar metas                       |
| Basado en utilidad      | Elige entre metas con grados de preferencia         |
| De aprendizaje          | Mejora su comportamiento con la experiencia         |

*** PEAS (ejemplo: taxi autónomo)
- *Performance:* seguridad, rapidez, comodidad, legalidad.
- *Environment:* calles, tráfico, peatones, clima.
- *Actuators:* volante, acelerador, freno, señales.
- *Sensors:* cámaras, LIDAR, GPS, odometría.

*** Propiedades del entorno (para elegir técnica)
| Propiedad           | Ejemplos                                              |

| Discreto / continuo | Tablero vs conducción continua                        |
| Un agente / multi  | Solitario vs subastas, tráfico                        |

*** Sistemas multiagente (SMA)
- Varios agentes en un *mismo entorno*; pueden cooperar, competir o ser neutros.
- *Cooperación:* robots en almacén, enjambres.
- *Competencia:* subastas, juegos, mercados.

** 1.8 El papel de la heurística

*** ¿Qué es una heurística?
Una *heurística* es una regla práctica, atajo o función estimadora que *guía la búsqueda* hacia soluciones prometedoras sin garantizar optimalidad ni completitud en todos los casos.

*Motivación:* el espacio de estados crece exponencialmente (*explosión combinatoria*). Explorar todo es inviable.

*** Propiedades deseables
- *Admisibilidad (A*):* h(n) nunca sobreestima el costo real → solución óptima si existe.
- *Consistencia (monotonicidad):* h(n) ≤ costo(n,n') + h(n') → facilita optimalidad.
- *Información:* cuanto más cerca de la realidad, menos nodos se expanden (pero más costosa de calcular).

*** Búsqueda ciega vs informada vs local
| Tipo        | Usa costo/heurística | Ejemplos                    |
|-------------+----------------------+-----------------------------|
| Ciega       | No (solo estructura) | BFS, DFS, costo uniforme    |
| Informada   | Sí (h)               | A*, greedy best-first        |
| Local       | Mejora iterativa     | Hill climbing, simulated annealing |

** 1.8.1 Algoritmos de exploración de alternativas

*** Modelo de problema de búsqueda
- *Estado:* configuración del mundo.
- *Estado inicial* y *estados meta*.
- *Acciones* aplicables y *función de transición*.
- *Función de costo de camino* g(n) (suma de costos desde el inicio).
- *Árbol / grafo de búsqueda:* nodos = estados; aristas = acciones.

*** Búsqueda en anchura (BFS)
- Expande por *niveles* (frontera FIFO).
- *Completa* si el factor de ramificación es finito.
- *Óptima* si todos los pasos tienen el mismo costo.
- *Memoria:* alta (guarda toda la frontera).

*** Búsqueda en profundidad (DFS)
- Expande el nodo más profundo primero (pila).
- *Memoria:* baja (solo camino actual + frontera).
- Riesgo de ciclos → usar conjunto *visited* o límite de profundidad (IDS).

*** Costo uniforme (UCS / Dijkstra en grafo de estados)
- Expande el nodo de *menor g(n)* (cola de prioridad).
- Óptima con costos no negativos.
- Sin heurística: búsqueda “ciega” informada solo por costos acumulados.

*** Greedy best-first
- Expande el nodo con menor h(n) (solo heurística).
- Rápida pero *no garantiza* optimalidad ni completitud en todos los grafos.

*** Ejemplo: laberinto en Python (BFS)

#+begin_src python
from collections import deque

def bfs(start, goal, neighbors):
    """neighbors(state) -> iterable de estados alcanzables."""
    queue = deque([(start, [start])])
    visited = {start}
    while queue:
        state, path = queue.popleft()
        if state == goal:
            return path
        for nxt in neighbors(state):
            if nxt not in visited:
                visited.add(nxt)
                queue.append((nxt, path + [nxt]))
    return None

# Grafo de ejemplo: 0 -- 1 -- 2
#                      \     /
#                        3
graph = {
    0: [1, 3],
    1: [0, 2],
    2: [1, 3],
    3: [0, 2],
}
print("BFS 0→2:", bfs(0, 2, lambda s: graph[s]))
#+end_src

** 1.8.2 Algoritmo A*

*** Idea
A* combina:
- g(n) = costo acumulado desde el inicio.
- h(n) = estimación del costo restante hasta la meta.

*f(n) = g(n) + h(n)*

Expande siempre el nodo con *menor f(n)* en una cola de prioridad.

*** Pseudocódigo (conceptual)
#+begin_example
OPEN ← {nodo_inicial}
g(inicio) ← 0
mientras OPEN no vacío:
    n ← nodo en OPEN con menor f(n) = g(n) + h(n)
    si n es meta: reconstruir camino y terminar
    para cada sucesor n' de n:
        g' ← g(n) + costo(n, n')
        si g' < g(n') o n' no visitado:
            g(n') ← g'
            f(n') ← g' + h(n')
            padre(n') ← n
            añadir n' a OPEN
#+end_example

*** Condiciones para optimalidad
- El grafo es finito o hay criterio de corte.
- h es *admisible* (no sobreestima).
- Con h *consistente*, A* no re-expande innecesariamente.

*** Ejemplo: A* en cuadrícula (Manhattan)

#+begin_src python
import heapq

def manhattan(a, b):
    return abs(a[0] - b[0]) + abs(a[1] - b[1])

def astar(start, goal, neighbors, cost=1):
    """
    start, goal: tuplas (fila, col)
    neighbors(pos) -> lista de posiciones válidas
    """
    open_heap = []
    heapq.heappush(open_heap, (manhattan(start, goal), 0, start))
    g_score = {start: 0}
    came_from = {}

    while open_heap:
        f, g, current = heapq.heappop(open_heap)
        if current == goal:
            path = [current]
            while current in came_from:
                current = came_from[current]
                path.append(current)
            return list(reversed(path))

        for nxt in neighbors(current):
            new_g = g_score[current] + cost
            if nxt not in g_score or new_g < g_score[nxt]:
                g_score[nxt] = new_g
                came_from[nxt] = current
                h = manhattan(nxt, goal)
                heapq.heappush(open_heap, (new_g + h, new_g, nxt))
    return None

# Cuadrícula 3x3 sin obstáculos
def grid_neighbors(pos):
    r, c = pos
    moves = []
    for dr, dc in [(-1,0),(1,0),(0,-1),(0,1)]:
        nr, nc = r + dr, c + dc
        if 0 <= nr < 3 and 0 <= nc < 3:
            moves.append((nr, nc))
    return moves

print("A* (0,0)→(2,2):", astar((0,0), (2,2), grid_neighbors))
#+end_src

*** Cuándo usar A*
- Mapas, planificación de rutas, puzzles con costo en aristas.
- Cuando existe una h admisible razonable (distancia euclidiana, Manhattan, patrón desajustado en 8-puzzle).

** 1.8.3 Algoritmos de búsqueda local

*** Idea
En lugar de explorar sistemáticamente todo un árbol, la *búsqueda local* parte de una *solución candidato* y la va *mejorando* iterativamente según una *función objetivo* (o energía).

Útil cuando:
- El camino importa menos que el *estado final* (optimización, CSP relajados).
- El espacio es enorme pero hay estructura local.

*** Hill climbing (escalada de colina)
1. Estado actual s.
2. Evaluar vecinos; moverse al vecino con *mejor* valor.
3. Parar si no hay mejora (máximo local).

*Problema:* máximos locales, mesetas, crestas.

*** Variantes
| Algoritmo              | Idea                                           |
|------------------------+------------------------------------------------|
| Steepest ascent        | Mejor vecino en cada paso                      |
| Stochastic hill climb  | Elige aleatoriamente entre mejoras             |
| Random restart         | Reinicia desde puntos aleatorios               |
| Simulated annealing    | Acepta empeoras con probabilidad decreciente   |
| Genetic algorithms     | Población, cruce, mutación, selección          |

*** Ejemplo: hill climbing (maximizar función 1D discreta)

#+begin_src python
import random

def hill_climbing(f, start, neighbors, max_steps=100):
    current = start
    value = f(current)
    for _ in range(max_steps):
        nbrs = list(neighbors(current))
        if not nbrs:
            break
        best = max(nbrs, key=f)
        best_val = f(best)
        if best_val <= value:
            break  # máximo local
        current, value = best, best_val
    return current, value

# Función con máximo local en x=2 y global en x=8 (dominio 0..10)
def f(x):
    return -(x - 8) ** 2 + 64 if x >= 5 else -(x - 2) ** 2 + 10

def nbrs(x):
  for d in (-1, 1):
      if 0 <= x + d <= 10:
          yield x + d

for trial in range(3):
    s0 = random.randint(0, 10)
    s, v = hill_climbing(f, s0, nbrs)
    print(f"inicio={s0} → máximo local en x={s}, f={v}")
#+end_src

** 1. El problema de los maridos celosos

*** Enunciado
Tres parejas (tres maridos y sus tres esposas) deben cruzar un río usando un bote con capacidad máxima para *dos personas*.

*Restricción:* ninguna mujer puede quedarse en presencia de *otro* hombre a menos que su *propio marido* también esté presente (en orillas y en el bote).

*Objetivo:* secuencia de viajes *mínima* para que las seis personas crucen sin violar la restricción.

*** Modelado del espacio de estados (sin enumerar todo a ciegas)

**** Variables
- Conjunto =L= (orilla izquierda) y =R= (orilla derecha), con =L ⊔ R = {H1,W1,H2,W2,H3,W3}=.
- Posición del bote: =bote ∈ {izquierda, derecha}=.
- *Estado:* =(personas_en_R, bote_izquierda)= (equivalente a saber quién está en cada lado).

**** Acciones
Desde la orilla donde está el bote, elegir 1 o 2 personas que cruzan.

**** Predicado lógico de seguridad (poda)
Para cada orilla =X=:

#+begin_example
segura(X) ≡ ∀ mujer Wi ∈ X, ∀ hombre Hj ∈ X con j≠i:
              (Hj ∈ X) → (Hi ∈ X)

Si Wi está con un hombre que NO es su marido, su marido Hi también debe estar en X.
#+end_example

*Poda:* solo se expanden sucesores con =segura(L) ∧ segura(R)=.

**** Tamaño del espacio
- Bruto: =2^6 × 2 = 128= estados.
- Con poda lógica: el BFS explora ~*40* estados (orden de magnitud), no un árbol exponencial de secuencias de movimientos.

*** Razonamiento guiado (lógica antes que fuerza bruta)

1. *Mover dos esposas primero* es seguro (no hay hombres en la orilla de llegada).
2. *Regresar una esposa* devuelve el bote sin dejar a una mujer con hombres ajenos sin su marido.
3. *Dos maridos* pueden cruzar cuando en la otra orilla sus esposas ya están o la composición es segura.
4. La tercera pareja (=H3,W3=) cruza cuando el tablero lo permite.
5. Los últimos viajes reacomodan a quien debe regresar con el bote.

*** Solución óptima: *9 viajes*

| Paso | Viaje      | Izquierda (L)   | Derecha (R)        | Bote |
|------+------------+-----------------+--------------------+------|
| 0    | —          | H1 W1 H2 W2 H3 W3 | —                | L    |
| 1    | W1, W2 →   | H1 H2 H3 W3     | W1 W2              | R    |
| 2    | W1 ←       | H1 W1 H2 H3 W3  | W2                 | L    |
| 3    | H1, H2 →   | H3 W3           | H1 H2 W1 W2        | R    |
| 4    | H1, W1 ←   | H1 W1 H3 W3     | H2 W2              | L    |
| 5    | H3, W3 →   | H1 W1           | H2 W2 H3 W3        | R    |
| 6    | H2, W2 ←   | H1 W1 H2 W2     | H3 W3              | L    |
| 7    | H1, H2 →   | W1 W2           | H1 H2 H3 W3        | R    |
| 8    | H1, W1 ←   | H1 W1           | H2 W2 H3 W3        | L    |
| 9    | H1, W1 →   | —               | H1 W1 H2 W2 H3 W3  | R ✓  |

*Paso crítico 7:* en R están =H1,H2,H3,W3=. =W3= convive con =H1= y =H2= porque su marido =H3= también está → *válido*.

** 2. El puente y la linterna (optimización en 17 minutos)

*** Enunciado
Cuatro personas deben cruzar un puente de noche. El puente soporta *dos* personas. Hay *una* linterna.

| Persona | Tiempo |
|---------+--------|
| A       | 1 min  |
| B       | 2 min  |
| C       | 5 min  |
| D       | 10 min |

Dos personas cruzan a la velocidad de la *más lenta*. La linterna debe volver para el siguiente cruce.

*Objetivo:* todos al otro lado en exactamente *17 minutos*.

*** Modelado (estado acotado)

**** Estado =(C, ℓ, t)=
- =C ⊆ {A,B,C,D}=: quienes ya cruzaron.
- =ℓ=: linterna en lado *cerca* o *lejos*.
- =t=: tiempo acumulado (minutos).

**** Poda (evita explosión)
1. Si =t > 17= → *cortar rama*.
2. *Dominancia:* para igual =(C, ℓ)=, quedarse solo con el camino de menor =t=.
3. Espacio total acotado: ≈ 16 × 2 × 18 estados (minúsculo frente a enumerar todas las secuencias de pares).

*** Razonamiento lógico (sin BFS)

*Lema informal:* los lentos =C= y =D= deben cruzar juntos *como máximo una vez* en la solución óptima (cuesta 10 min). Si cada uno cruzara por separado con escolta, se pagan más de 10 min solo en esos cruces.

*Estrategia ganadora:*
1. Mandar primero los dos rápidos (=A+B=, 2 min).
2. Devolver el *más rápido* (=A=, 1 min).
3. Cruzar los dos lentos juntos (=C+D=, 10 min) — el gasto inevitable.
4. Devolver alguien moderado (=B=, 2 min) — mejor que dos vueltas de =A= si eso habilita el cierre.
5. Cerrar con =A+B= (2 min).

Total: 2+1+10+2+2 = *17*.

*** Solución paso a paso

| Paso | Acción       | Δt | Acumulado | En lado lejano |
|------+--------------+----+-----------+----------------|
| 1    | A + B →      | 2  | 2         | A, B           |
| 2    | A ←          | 1  | 3         | B              |
| 3    | C + D →      | 10 | 13        | B, C, D        |
| 4    | B ←          | 2  | 15        | C, D           |
| 5    | A + B →      | 2  | *17*      | A, B, C, D ✓   |

** 3. Flavio Josefo y el círculo de la muerte

*** Historia
*Flavio Josefo* relata que, sitiado en Yodfat con 40 compañeros (41 hombres en total), convinieron en morir por sorteo circular antes que rendirse. Josefo calculó la posición segura y sobrevivió — de ahí el *problema de Josefo*.

*** Enunciado matemático
- =n= personas en círculo, numeradas =1 … n=.
- Se cuenta en círculo y se elimina cada =k=-ésima persona *viva*.
- Se repite hasta quedar *una*.
- *¿Qué posición inicial garantiza sobrevivir?* =J(n, k)=.

*** ¿Por qué no expandir el árbol completo?
Un árbol de “quién se elimina en cada ronda” tiene ramificación variable y profundidad =n-1=; para =n= grande es innecesario.

| Enfoque              | Complejidad | Cuándo usarlo              |
|----------------------+-------------+----------------------------|
| Simulación en lista  | O(n²)       | Cualquier =k=, =n= moderado |
| Recurrencia          | O(n)        | Programación dinámica      |
| Fórmula cerrada k=2  | O(log n)    | “Cada segundo” (clásico)   |

*** Caso k = 2 (eliminar cada segundo)

**** Fórmula cerrada
Sea =m = ⌊log₂ n⌋= y =2^m ≤ n < 2^(m+1)=:

\[ J(n, 2) = 2 \cdot (n - 2^m) + 1 \]

**** Intuición
Cada ronda con =k=2= elimina la mitad (aprox.) de los vivos; la posición sobreviviente se obtiene “reflejando” el índice respecto a la mayor potencia de 2 ≤ =n=.

**** Historia: n = 41 soldados
- =2^5 = 32 ≤ 41 < 64= → =m = 5=.
- =J(41, 2) = 2·(41 - 32) + 1 = 19=.

*Josefo debía situarse en la posición 19* (contando desde 1) si el sorteo eliminaba cada segundo.

*** Ejemplo manual: n = 7, k = 2

Con la convención estándar (=J(1,2)=1=, =J(2n)=2J(n)-1=, =J(2n+1)=2J(n)+1=):

#+begin_example
J(1)=1, J(2)=1, J(3)=3, J(4)=1, J(5)=3, J(6)=5, J(7)=7
#+end_example

Fórmula potencias de 2: =2·(7-4)+1 = 7=. La posición segura es la *7* (contando desde 1).

*** Caso general k (algoritmo iterativo O(n))

#+begin_example
J ← 0
para i = 1 … n:
    J ← (J + k) mod i
respuesta (1-indexada) = J + 1
#+end_example

Este algoritmo evita simular el círculo completo (=O(n²)=) y coincide con la fórmula cerrada cuando =k=2=.


*** Síntesis: lógica vs explosión combinatoria

| Problema        | Truco principal                          | Viajes / respuesta      |
|-----------------+------------------------------------------+-------------------------|
| Maridos celosos | Predicado =segura(orilla)= poda BFS      | *9* viajes mínimos      |
| Puente          | C+D juntos una vez; poda =t>17=          | *17* minutos            |
| Josefo (k=2)    | Fórmula con potencias de 2; O(n) rec.    | =J(41,2)=*19*           |

** Parte 1 — ¿Qué es un estado? (la “fotografía” del río)

En cada momento anotas *tres cosas*:

[ Quién está a la IZQUIERDA | Quién está a la DERECHA ]   BOTE

Ejemplo inicio:
  [ A L C M | — ]   Bote en izquierda ◄

  [ — | A L C M ]   Bote en derecha ►
#+end_example

** Parte 2 — El espacio de estados (el mapa completo del camino bueno)

*** Los 8 estados del camino más corto (7 cruces)

Usamos códigos E0, E1, … E* para no repetir dibujos.

| Código | Izquierda | Derecha | Bote | Qué acaba de pasar |
|--------+-----------+---------+------+--------------------|
| E0     | A L C M   | —       | ◄    | (inicio)           |
| E1     | L M       | A C     | ►    | A+C cruzan →       |
| E2     | A L M     | C       | ◄    | A regresa ←        |
| E3     | M         | A L C   | ►    | A+L cruzan →       |
| E4     | A C M     | L       | ◄    | A+C regresan ←     |
| E5     | C         | A L M   | ►    | A+M cruzan →       |
| E6     | A C       | L M     | ◄    | A regresa ←        |

*** Dibujo del camino (solo flechas hacia adelante)

#+begin_example
E0 ──A+C──► E1 ──A──► E2 ──A+L──► E3 ──A+C──► E4 ──A+M──► E5 ──A──► E6 ──A+C──► E*
 ◄          ►         ◄          ►          ◄          ►         ◄          ►
#+end_example

*** ¿Por qué no hay más estados en el camino bueno?
Porque en cada paso solo hay *pocas* opciones legales; muchas combinaciones se descartan al instante (coyote con cabra, etc.). El espacio *total* válido es pequeño (unas 16 fotografías); no hace falta dibujarlas todas si A* te guía.

*** Ramas que se cortan desde E0 (ejemplo de poda)

Desde E0 el arriero está a la izquierda. Prueba mental:

| Cruce intentado | ¿Se puede? | Motivo                              |
|-----------------+------------+-------------------------------------|
| A solo          | NO         | Quedan L y C solos a la izquierda   |
| A + L           | NO         | Quedan C y M solos (cabra + maíz)   |
| A + C           | SÍ → E1    | Ambas orillas seguras               |
| A + M           | NO         | Quedan L y C solos (coyote + cabra) |

*Sin A*, ya eliminaste 3 de 4 movimientos. Eso es *poda*: no pierdes tiempo en fotografías imposibles.

** Parte 3 — g, h y f 

*** g = “¿Cuántos cruces ya hice?”
- Empiezas en E0 con *g = 0*.
- Cada cruce del bote: *g sube 1*.
- En E3 llevas *g = 3* (tres viajes hechos).

Regla que usamos (fácil de contar a mano):

#+begin_example
Mira la orilla DERECHA.
Cuenta cuántas de estas piezas AÚN NO están ahí:
  □ Coyote (L)
  □ Cabra  (C)
  □ Maíz   (M)

#+end_example

Ejemplos:

#+end_example

| Estado | g | h | f = g+h |

*  Redes Neuronales y Backpropagation

** ¿Qué es una neurona artificial?

Una neurona es una calculadora diminuta. Recibe números de entrada, cada entrada
tiene un *peso* (importancia), los combina y produce una salida.

#+begin_src text
   x1 ──(peso w1)──┐
                    │
   x2 ──(peso w2)──┼──► [ suma: z = w1·x1 + w2·x2 + b ] ──► [ activación ] ──► salida
                    │
   x3 ──(peso w3)──┘
                    (b = "bias" o sesgo, un ajuste extra)
#+end_src

- Los *pesos* (w) dicen cuánta importancia tiene cada entrada.
- El *bias* (b) es un número extra que permite mover el resultado.
- La *función de activación* dobla el resultado para que la red aprenda cosas no
  lineales (curvas, no solo líneas rectas). Ejemplos: sigmoide, ReLU.

*Aprender = encontrar los mejores pesos y biases.* Eso es todo lo que hace una red.

** La analogía de las perillas

Imagina una mesa de mezclas de música con muchas perillas (los pesos). Al principio
están en posiciones aleatorias y la música suena horrible (predicciones malas).

El *backpropagation* es el procedimiento para saber, para CADA perilla, *hacia qué
lado y cuánto* girarla para que la música (predicción) mejore un poquito. Repites
miles de veces y al final suena bien.

** Las 4 fases del entrenamiento

#+begin_src text
   ┌─────────────────────────────────────────────────────────┐
   │  1. FORWARD  (hacia adelante)                             │
   │     Los datos entran y la red produce una predicción ŷ    │
   ├─────────────────────────────────────────────────────────┤
   │  2. LOSS     (calcular el error)                          │
   │     Comparamos ŷ con el real y usando, p.ej., el MSE      │
   ├─────────────────────────────────────────────────────────┤
   │  3. BACKWARD (backpropagation)                            │
   │     Calculamos cuánta "culpa" tiene cada peso del error   │
   ├─────────────────────────────────────────────────────────┤
   │  4. UPDATE   (ajuste de pesos, gradiente descendente)     │
   │     Movemos cada peso un poquito para reducir el error    │
   └─────────────────────────────────────────────────────────┘
                         ↑                          │
                         └──────── repetir ─────────┘
#+end_src

** El corazón: el gradiente y la pendiente

Imagina que estás en una montaña con los ojos vendados y quieres bajar al valle
(donde el error es mínimo). ¿Qué haces? Tanteas con el pie hacia dónde baja el
suelo y das un paso en esa dirección. Repites hasta llegar abajo.

- La *montaña* es la función de error (el error depende de los pesos).
- La *pendiente* bajo tus pies es el *gradiente* (la derivada).
- El *tamaño del paso* es la *tasa de aprendizaje* (learning rate).

La regla mágica del *gradiente descendente*:

#+begin_src text
   nuevo_peso = viejo_peso − (learning_rate × gradiente)
#+end_src

- El *gradiente* dice "hacia dónde SUBE el error". Por eso lo restamos (queremos BAJAR).
- El *learning rate* (ej. 0.01) controla el tamaño del paso:
  - Muy grande → das saltos enormes y te pasas del valle (no converge).
  - Muy pequeño → tardas una eternidad en bajar.

** ¿Por qué se llama "back"-propagation?

Porque la culpa del error se reparte *hacia atrás*, desde la salida hasta la entrada,
usando la *regla de la cadena* del cálculo (derivadas encadenadas).

#+begin_src text
   Entrada → Capa 1 → Capa 2 → Salida → ERROR
                                          │
   Entrada ← Capa 1 ← Capa 2 ← Salida ←───┘
   (el error "viaja hacia atrás" repartiendo culpa a cada peso)
#+end_src

Intuición de la regla de la cadena: si el error depende de la salida, la salida
depende de la capa 2, y la capa 2 depende de un peso... entonces multiplicamos esos
"cuánto afecta a" para saber cuánto ese peso afecta al error final.

** Ejemplo súper simple a mano (1 peso)

Tenemos una relación =ŷ = w · x=. Queremos que con =x = 2= la salida sea =y = 10=
(o sea, el peso ideal es w = 5). Empezamos con un peso malo =w = 1= y learning_rate = 0.1.

Error usado: =E = (y − ŷ)²=. El gradiente de E respecto a w es:

#+begin_src text
   dE/dw = 2·(ŷ − y)·x           (esto sale de la regla de la cadena)
#+end_src

*Iteración 1:*

#+begin_src text
   ŷ = w·x = 1·2 = 2
   error = (10 − 2)² = 64
   gradiente = 2·(2 − 10)·2 = 2·(−8)·2 = −32
   w_nuevo = 1 − 0.1·(−32) = 1 + 3.2 = 4.2
#+end_src

*Iteración 2:*

#+begin_src text
   ŷ = 4.2·2 = 8.4
   error = (10 − 8.4)² = 2.56        ← ¡bajó muchísimo!
   gradiente = 2·(8.4 − 10)·2 = −6.4
   w_nuevo = 4.2 − 0.1·(−6.4) = 4.84
#+end_src

El peso va de 1 → 4.2 → 4.84 → ... acercándose a 5, y el error baja de 64 → 2.56 → ...
*Eso es aprender.* Backpropagation solo hace esto para MILES de pesos a la vez.

** Resumen de la Parte 2

- Una red neuronal es un montón de neuronas con *pesos* ajustables.
- *Forward* = calcular la predicción. *Loss* = medir el error (MSE u otro).
- *Backpropagation* = repartir la culpa del error a cada peso (con derivadas).
- *Gradiente descendente* = mover cada peso un pasito para bajar el error.
- Repetir muchas *épocas* hasta que el error sea pequeño.

* Error Cuadrático Medio


Este apartado  explica,  conceptos fundamentales del Machine Learning (aprendizaje automático):

1. El *Error Cuadrático Medio* (MSE): cómo medimos qué tan "mal" lo hace un modelo.
2. El *Backpropagation*: cómo una red neuronal ajusta sus "pesos" para mejorar.
3. La *Regresión Lineal Simple*: predecir un número con una línea recta.
4. La *Regresión Logística*: predecir "sí o no" (clasificación).
5. Los *Árboles de Decisión* (Decision Trees): tomar decisiones con preguntas.

La idea central de TODO el machine learning es esta:

#+begin_src text
   MODELO  →  hace una predicción  →  la comparamos con la realidad
                                            │
                                            ▼
                                 ¿Cuánto se equivocó?  (ERROR)
                                            │
                                            ▼
                          Ajustamos el modelo para equivocarse MENOS
                                            │
                                            └──── (repetir muchas veces)
#+end_src

Todo lo demás son detalles de *cómo* medir el error y *cómo* ajustar el modelo.

--------------------------------------------------------------------------------

** Parte 1: El Error Cuadrático Medio (MSE)

*** La idea con una analogía

Imagina que eres un profesor de tiro con arco. Cada alumno dispara y tú quieres
saber qué tan bueno es. No basta con mirar una flecha: quieres un *número* que
resuma qué tan lejos del centro queda en promedio.

El *Error Cuadrático Medio* (Mean Squared Error, MSE) es justo eso: un número que
resume qué tan lejos están las predicciones de un modelo respecto a la realidad.

*** ¿Por qué "cuadrático"?

Supón que tu modelo predice y la realidad es distinta. La diferencia se llama *error*:

#+begin_src text
   error = valor_real − valor_predicho
#+end_src

Problema: algunos errores son positivos (+3) y otros negativos (−3). Si los sumas
tal cual, ¡se cancelan y parece que no hay error! Solución: *elevar al cuadrado*.

- Al elevar al cuadrado, todo se vuelve positivo: (−3)² = 9, (+3)² = 9.
- Además, castiga MÁS los errores grandes: un error de 10 pesa 100, pero uno de 2
  solo pesa 4. Es decir, "equivocarse mucho" duele mucho más.

*** La fórmula (no te asustes)

#+begin_src text
          1     n
   MSE = ───  · Σ  (yᵢ − ŷᵢ)²
          n    i=1
#+end_src

Traducción palabra por palabra:

- =n=    → cuántos datos tienes (ej. 5 alumnos).
- =yᵢ=   → el valor REAL del dato i (la verdad).
- =ŷᵢ=   → el valor PREDICHO por el modelo para el dato i ("y sombrero").
- =(yᵢ − ŷᵢ)²= → el error de ese dato, elevado al cuadrado.
- =Σ=    → "suma todos" (la letra griega sigma = sumatoria).
- =(1/n)·= → divide entre la cantidad de datos = saca el PROMEDIO.

En cristiano: *"resta real menos predicho, elévalo al cuadrado, hazlo con todos,
súmalos y saca el promedio"*.

*** Ejemplo numérico a mano

Un modelo predice el precio de 3 casas (en miles):

| Casa | Real (y) | Predicho (ŷ) | Error (y−ŷ) | Error² |
|------+----------+--------------+-------------+--------|
|    1 |      200 |          190 |          10 |    100 |
|    2 |      150 |          160 |         −10 |    100 |
|    3 |      300 |          280 |          20 |    400 |

Paso a paso:

#+begin_src text
   Suma de errores²  = 100 + 100 + 400 = 600
   MSE = 600 / 3 = 200
#+end_src

El MSE es *200*. ¿Es bueno o malo? Por sí solo no dice mucho; sirve para COMPARAR:
si otro modelo da MSE = 50, ese otro es mejor (se equivoca menos).

Nota: a veces se usa el *RMSE* (raíz del MSE) para volver a las unidades originales:
√200 ≈ 14.1 miles de dólares de error típico.


*Idea clave*: el MSE es la "nota de examen" del modelo. Cuanto MÁS BAJO, mejor.
Entrenar un modelo = *buscar los parámetros que hacen el MSE lo más pequeño posible*.

* Descenso por Gradiente: Teoría y Casos Prácticos


Si el tutorial anterior te explicó *qué* es el MSE, el backpropagation y varios modelos,
este se centra en el *motor* que hace que casi todos aprendan:

#+begin_src text
   DESCENSO POR GRADIENTE (Gradient Descent)
   = "ir bajando paso a paso hasta encontrar el mejor ajuste"
#+end_src

En machine learning casi siempre pasa lo mismo:

1. Tienes un modelo con parámetros (pesos, pendiente, bias...).
2. Defines una *función de error* (loss) que dice qué tan mal predice.
3. Usas *descenso por gradiente* para mover esos parámetros y *bajar* el error.

No necesitas memorizar fórmulas complicadas. La idea es una sola:

#+begin_src text
   Estás en una montaña (error alto).
   Miras hacia dónde baja el terreno (gradiente).
   Das un paso en esa dirección (actualizas el parámetro).
   Repites hasta llegar al valle (error mínimo).
#+end_src

Este tutorial explica eso *desde cero*, con ejemplos a mano y código Python que puedes
correr. Al final sabrás *por qué* funciona, *cuándo* falla y *cómo* se usa en la vida real.

** Parte 1: La intuición

*** Analogía 1: bajar una montaña con los ojos vendados

Imagina que estás en lo alto de una montaña y quieres llegar al punto más bajo (el
valle). Tienes los ojos vendados. ¿Qué haces?

#+begin_src text
   1. Tanteas con el pie: ¿hacia dónde baja más el suelo?
   2. Das UN paso en esa dirección.
   3. Repites desde el nuevo punto.
#+end_src

- La *altura* donde estás = el *error* del modelo.
- La *dirección de la pendiente* = el *gradiente*.
- El *tamaño del paso* = la *tasa de aprendizaje* (learning rate, α).

No necesitas ver toda la montaña. Solo necesitas saber, *en el punto donde estás*,
hacia dónde sube más rápido... para ir en la dirección *contraria* (bajar).

*** Analogía 2: afinar una perilla de volumen

Tienes una perilla y un medidor que dice "qué tan mal suena la música". Giras la
perilla un poquito en la dirección que *baja* el medidor. Repites.

- Perilla = parámetro del modelo (ej. peso w).
- Medidor = función de error.
- Giro pequeño = un paso de gradiente descendente.

*** La regla de oro (memorízala)

#+begin_src text
   parámetro_nuevo = parámetro_viejo − α · gradiente
#+end_src

- =gradiente= → "si aumento un poquito este parámetro, ¿el error sube o baja y cuánto?"
- Restamos porque queremos *bajar* el error (ir contra la subida).
- =α= (alpha) → qué tan grande es cada paso. Típico: 0.001, 0.01, 0.1 (depende del problema).

*** ¿Qué es el gradiente, en criollo?

El *gradiente* es la *derivada* (pendiente) de la función de error respecto al parámetro.

#+begin_src text
   Si error = E(w)

   gradiente = dE/dw = "cuánto cambia E si w cambia un poquito"
#+end_src

- Gradiente *positivo* → si aumentas w, el error *sube*. Entonces debes *disminuir* w.
- Gradiente *negativo* → si aumentas w, el error *baja*. Entonces debes *aumentar* w.

La fórmula =parámetro − α·gradiente= hace eso automáticamente en ambos casos.

--------------------------------------------------------------------------------

** Caso práctico 1 — Encontrar el mínimo de una parábola (a mano)

El ejemplo más simple del mundo: minimizar

#+begin_src text
   f(w) = w²
#+end_src

Sabemos que el mínimo está en =w = 0= (porque 0² es lo más bajo). Pero *olvídalo* un
momento: vamos a llegar ahí solo con descenso por gradiente, como si no supiéramos la respuesta.

*** Paso 1: calcular el gradiente

La derivada de =w²= es =2w=:

#+begin_src text
   gradiente = df/dw = 2w
#+end_src

*** Paso 2: elegir punto de partida y learning rate

- Empezamos en =w = 4= (lejos del mínimo).
- =α = 0.1=

*** Paso 3: iterar (tabla a mano)

| Iteración | w   | f(w)=w² | gradiente=2w | w_nuevo = w − 0.1·2w |
|-----------+-----+---------+--------------+----------------------|
|         0 | 4.0 |    16.0 |          8.0 | —                    |
|         1 | 3.2 |   10.24 |          6.4 | 4 − 0.8 = 3.2        |
|         2 | 2.56|    6.55 |         5.12 | 3.2 − 0.64 = 2.56    |
|         3 | 2.05|    4.20 |         4.10 | ...                  |
|       ... | ... |     ... |          ... | se acerca a 0        |

Cada fila: calculas gradiente en w actual, restas =0.1 × gradiente=, y ese es el nuevo w.
El error =w²= *baja* en cada paso. Después de muchas iteraciones, w ≈ 0.

*** Código Python (mismo ejemplo)

#+begin_src python
def f(w):
    return w ** 2

def gradiente(w):
    return 2 * w

w = 4.0
alpha = 0.1

print(f"{'iter':>4}  {'w':>8}  {'f(w)':>10}  {'grad':>8}")
for i in range(15):
    g = gradiente(w)
    print(f"{i:4d}  {w:8.4f}  {f(w):10.4f}  {g:8.4f}")
    w = w - alpha * g

print(f"\nResultado final: w ≈ {w:.6f}  (el mínimo teórico es 0)")
#+end_src

*Lección*: con un gradiente correcto y un α razonable, el algoritmo converge al mínimo.

** Caso práctico 2 — El learning rate lo cambia TODO

El mismo problema =f(w)=w²=, mismo inicio =w=4=, pero distintos α.

*** α demasiado GRANDE (α = 1.0) — ¡explota!

#+begin_src text
   w=4  → grad=8  → w_nuevo = 4 − 1.0·8 = −4
   w=−4 → grad=−8 → w_nuevo = −4 − 1.0·(−8) = +4
   w=+4 → ... ¡rebota entre 4 y −4 para siempre!
#+end_src

El paso es tan grande que *te pasas* del valle y saltas al otro lado de la montaña.
Nunca converge. Es como bajar escaleras dando saltos de 10 metros.

*** α demasiado PEQUEÑO (α = 0.001) — muy lento

#+begin_src text
   w=4 → w_nuevo = 4 − 0.001·8 = 3.992
#+end_src

Funciona, pero necesitas *miles* de iteraciones para llegar cerca de 0. Es seguro pero
lento. En redes neuronales grandes, esto puede significar días de entrenamiento.

*** α "bien" (α = 0.1) — equilibrio

Baja rápido sin rebotar. En la práctica se prueba α en una escala logarítmica:
0.0001, 0.001, 0.01, 0.1... y se mira si el error baja estable.

*** Visualización en Python (tres curvas de aprendizaje)

#+begin_src python
import numpy as np
import matplotlib.pyplot as plt

def descenso(w0, alpha, pasos=30):
    w = w0
    historial = [w]
    for _ in range(pasos):
        w = w - alpha * (2 * w)
        historial.append(w)
    return historial

plt.figure(figsize=(8, 4))
for alpha, estilo in [(1.0, "r--"), (0.1, "g-"), (0.001, "b-")]:
  hs = descenso(4.0, alpha)
  plt.plot(hs, label=f"α={alpha}")
plt.axhline(0, color="gray", linestyle=":", alpha=0.5)
plt.xlabel("Iteración")
plt.ylabel("w")
plt.title("Mismo problema, distinto learning rate")
plt.legend()
plt.grid(True, alpha=0.3)
plt.savefig("/tmp/descenso_lr.png", dpi=120)
print("Gráfica guardada en /tmp/descenso_lr.png")
#+end_src

*Regla práctica*: si el error *oscila* o se vuelve NaN → α muy alto. Si casi no baja → α
muy bajo o pocos pasos.

** Caso práctico 3 — Regresión lineal con descenso por gradiente

Aquí ya es *machine learning de verdad*: ajustar una recta =ŷ = m·x + b= a datos reales.

*** Los datos (precio de pizzas según tamaño)

| Tamaño (pulgadas) x | Precio ($) y |
|---------------------+--------------|
|                   6 |           7  |
|                   8 |           9  |
|                  10 |          13  |
|                  12 |          15  |
|                  14 |          17  |

Queremos encontrar =m= y =b= que minimicen el MSE.

*** Función de error (MSE)

Para =n= puntos:

#+begin_src text
          1     n
   MSE = ───  · Σ  (yᵢ − (m·xᵢ + b))²
          n    i=1
#+end_src

*** Gradientes (de dónde salen los pasos)

Con un poco de cálculo (derivadas parciales):

#+begin_src text
   ∂MSE/∂m = (−2/n) · Σ  xᵢ · (yᵢ − ŷᵢ)
   ∂MSE/∂b = (−2/n) · Σ      (yᵢ − ŷᵢ)

   donde ŷᵢ = m·xᵢ + b  (predicción actual)
#+end_src

No tienes que derivarlo cada vez: en código solo implementas estas dos líneas.

*** Una iteración a mano (valores inventados al inicio)

Empezamos con =m = 0.5=, =b = 0=, =α = 0.01=.

Para el primer punto (x=6, y=7): ŷ = 0.5·6 + 0 = 3, error = 7−3 = 4.
Haces eso para los 5 puntos, sumas según las fórmulas de arriba, y obtienes
gradiente_m y gradiente_b. Luego:

#+begin_src text
   m_nuevo = m − α · gradiente_m
   b_nuevo = b − α · gradiente_b
#+end_src

Repite 500–2000 veces y m, b convergen a algo cercano a la recta óptima.

*** Código completo (regresión lineal desde cero)

#+begin_src python
import numpy as np

# Datos: tamaño de pizza (pulgadas) → precio ($)
X = np.array([6, 8, 10, 12, 14])
y = np.array([7, 9, 13, 15, 17])

m, b = 0.0, 0.0          # empezar "en cero" (recta plana)
alpha = 0.0001           # learning rate (datos pequeños → α pequeño)
n = len(y)
epocas = 2000

for epoca in range(epocas):
    y_pred = m * X + b
    error = y - y_pred

    grad_m = (-2 / n) * np.sum(X * error)
    grad_b = (-2 / n) * np.sum(error)

    m -= alpha * grad_m
    b -= alpha * grad_b

    if epoca % 400 == 0:
        mse = np.mean(error ** 2)
        print(f"época {epoca:4d}  m={m:.4f}  b={b:.4f}  MSE={mse:.4f}")

print(f"\nRecta aprendida: precio ≈ {m:.3f}·tamaño + {b:.3f}")
print(f"Predicción pizza 11\": ${m*11 + b:.2f}")
#+end_src

*** Comparar con la solución exacta (mínimos cuadrados)

#+begin_src python
from sklearn.linear_model import LinearRegression

modelo = LinearRegression()
modelo.fit(X.reshape(-1, 1), y)
print(f"sklearn: m={modelo.coef_[0]:.4f}, b={modelo.intercept_:.4f}")
# Debería ser muy parecido a lo que encontró el descenso por gradiente
#+end_src

*Lección*: el descenso por gradiente *aproxima* la misma solución que la fórmula
cerrada de mínimos cuadrados, pero funciona también cuando *no* hay fórmula cerrada
(redes neuronales, millones de parámetros).

** Caso práctico 4 — Clasificación con regresión logística

Mismo truco, otro error: en vez de MSE usamos *entropía cruzada* (log loss), y la
salida pasa por *sigmoide* para obtener probabilidad entre 0 y 1.

*** Datos: ¿aprobó el examen según horas de estudio?

| Horas x | Aprobó y (1=sí, 0=no) |
|---------+-----------------------|
|       1 |                   0   |
|       2 |                   0   |
|       3 |                   0   |
|       4 |                   1   |
|       5 |                   1   |
|       6 |                   1   |

Modelo: =p = sigmoide(m·x + b)=, con =sigmoide(z) = 1/(1+e^(−z))=.

*** Regla de actualización (simplificada)

Después de calcular predicciones =pᵢ= y errores =(yᵢ − pᵢ)=, los gradientes tienen
*la misma forma* que en regresión lineal, pero con predicciones ya pasadas por sigmoide:

#+begin_src text
   grad_m = (1/n) · Σ  xᵢ · (pᵢ − yᵢ)
   grad_b = (1/n) · Σ       (pᵢ − yᵢ)

   m ← m − α · grad_m
   b ← b − α · grad_b
#+end_src

*** Código

#+begin_src python
import numpy as np

def sigmoide(z):
    return 1 / (1 + np.exp(-z))

X = np.array([1, 2, 3, 4, 5, 6])
y = np.array([0, 0, 0, 1, 1, 1])

m, b = 0.0, 0.0
alpha = 0.1
n = len(y)

for epoca in range(3000):
    z = m * X + b
    p = sigmoide(z)
    error = p - y

    grad_m = (1 / n) * np.sum(X * error)
    grad_b = (1 / n) * np.sum(error)

    m -= alpha * grad_m
    b -= alpha * grad_b

print(f"m={m:.4f}, b={b:.4f}")
for horas in [2.5, 3.5, 4.5]:
    prob = sigmoide(m * horas + b)
    print(f"  {horas}h → P(aprobó)={prob:.1%}")
#+end_src

*Lección*: cambia la función de error y la salida, pero el *mecanismo* es idéntico:
calcular gradiente → restar α·gradiente → repetir.

** Batch, Stochastic y Mini-batch — ¿con cuántos datos calculo el gradiente?

Cada vez que actualizas los parámetros, necesitas un gradiente. La pregunta es: ¿usas
*TODOS* los datos, *UNO* solo, o un *grupo pequeño*?

*** Gradiente descendente por lotes (Batch GD)

#+begin_src text
   En CADA paso usas TODOS los ejemplos del dataset.
   Gradiente = promedio del gradiente de cada ejemplo.
#+end_src

- *Ventaja*: gradiente estable, camino suave hacia el mínimo.
- *Desventaja*: lento con millones de datos; cada paso recorre todo el dataset.

*** Gradiente descendente estocástico (SGD — Stochastic)

#+begin_src text
   En CADA paso usas UN solo ejemplo (elegido al azar).
   Actualizas m y b, luego otro ejemplo, luego otro...
#+end_src

- *Ventaja*: muchísimos pasos baratos; puede escapar de mínimos locales malos; online.
- *Desventaja*: trayectoria "ruidosa" (zigzag); un solo ejemplo puede engañar.

*** Mini-batch (el estándar en la práctica)

#+begin_src text
   En CADA paso usas un grupo pequeño (ej. 32, 64, 128 ejemplos).
   Compromiso entre estabilidad y velocidad.
#+end_src

PyTorch, TensorFlow y Keras usan mini-batch casi siempre. Cuando lees "batch size = 32",
significa: calculas el gradiente con 32 ejemplos, actualizas pesos, repites con los
siguientes 32.

*** Comparación visual (conceptual)

#+begin_src text
   Batch:      ●─────────────────────────────►  (línea suave al mínimo)
   SGD:        ●╱╲╱╲╱╲╱╲╱╲╱╲╱╲╱╲╱╲╱╲╱╲──►  (zigzag pero llega)
   Mini-batch: ●──╲╱─╲╱──╲╱───╲╱──────►  (entre ambos)
#+end_src

*** Mini-batch en código (regresión lineal)

#+begin_src python
import numpy as np

X = np.array([6, 8, 10, 12, 14])
y = np.array([7, 9, 13, 15, 17])
m, b = 0.0, 0.0
alpha = 0.001
batch_size = 2
epocas = 100

for epoca in range(epocas):
    # Mezclar índices cada época (buena práctica)
    indices = np.random.permutation(len(X))
    for inicio in range(0, len(X), batch_size):
        lote = indices[inicio : inicio + batch_size]
        xb, yb = X[lote], y[lote]
        n_batch = len(xb)

        y_pred = m * xb + b
        error = yb - y_pred
        grad_m = (-2 / n_batch) * np.sum(xb * error)
        grad_b = (-2 / n_batch) * np.sum(error)

        m -= alpha * grad_m
        b -= alpha * grad_b

print(f"Mini-batch: m={m:.4f}, b={b:.4f}")
#+end_src

**  Caso práctico 5 — Dos parámetros (m y b): superficie de error

Con un solo parámetro (w) es una *curva*. Con dos (m, b) el error es una *superficie*
(como un bowl, cuenco). El descenso por gradiente baja por esa superficie en 2D.

#+begin_src text
        MSE (error)
           ▲
           │     ╱╲
           │    ╱  ╲
           │   ╱    ╲   ← "cuenco"
           │  ╱  ★   ╲     ★ = mínimo (mejor m, b)
           │ ╱        ╲
           └──────────────► m
          ╱
         ╱ b
#+end_src

En cada paso actualizas *ambos* a la vez:

#+begin_src text
   m ← m − α · ∂MSE/∂m
   b ← b − α · ∂MSE/∂b
#+end_src

El gradiente en varias dimensiones es un *vector* de derivadas parciales: una por cada
parámetro. La regla sigue siendo la misma.

*** Código: ver cómo baja el MSE época a época

#+begin_src python
import numpy as np

X = np.array([6, 8, 10, 12, 14])
y = np.array([7, 9, 13, 15, 17])
m, b = 0.0, 0.0
alpha = 0.0001

mse_historial = []
for _ in range(2000):
    y_pred = m * X + b
    error = y - y_pred
    mse_historial.append(np.mean(error ** 2))

    grad_m = (-2 / len(y)) * np.sum(X * error)
    grad_b = (-2 / len(y)) * np.sum(error)
    m -= alpha * grad_m
    b -= alpha * grad_b

print("MSE al inicio:", mse_historial[0])
print("MSE al final: ", mse_historial[-1])
print("¿Bajó?", mse_historial[-1] < mse_historial[0])
#+end_src

Si imprimes =mse_historial= cada 100 épocas, verás una lista que *decrece*. Eso es
*aprender*: el modelo se equivoca cada vez menos.

** Problemas comunes y cómo reconocerlos

*** El error no baja nada

| Síntoma              | Causa probable              | Qué probar                    |
|----------------------+-----------------------------+-------------------------------|
| MSE plano desde el 0 | α demasiado pequeño         | Subir α (×10)                 |
| MSE = NaN o infinito | α demasiado grande          | Bajar α (÷10)                 |
| MSE baja y sube ola  | α alto o datos sin escalar  | Bajar α; normalizar X         |
| MSE baja y se estanca| Mínimo local o pocos pasos  | Más épocas; otro α            |

*** Normalización de datos (muy importante en la práctica)

Si una característica va de 0 a 1 y otra de 0 a 1 000 000, los gradientes se
deshbalancean y el descenso va mal. Solución habitual:

#+begin_src python
from sklearn.preprocessing import StandardScaler
# X_scaled = (X - media) / desviación estándar
# Entrena con X_scaled; muchas veces converge más rápido y estable
#+end_src

*** Mínimos locales (¿me quedo atrapado?)

En una parábola simple hay un solo mínimo. En redes neuronales la superficie es más
complicada. Aun así, en la práctica el descenso por gradiente con buen α, mini-batch
y muchas épocas suele encontrar soluciones *buenas* (no siempre la óptima global).

*** Cuándo parar de entrenar

- El error en *validación* deja de bajar (o sube) → posible *overfitting*; para ahí.
- El error en entrenamiento ya es suficientemente bajo para tu aplicación.
- Se acabó el presupuesto de tiempo/compute.

** Optimizadores — más allá del descenso "vanilla"

El descenso básico =θ ← θ − α·gradiente= funciona, pero la industria usa variantes
que adaptan el paso automáticamente:

| Optimizador | Idea en una frase                                      |
|-------------+--------------------------------------------------------|
| SGD         | El clásico; a veces con "momentum" (inercia al bajar)  |
| Adam        | Ajusta α por parámetro; muy usado por defecto          |
| RMSprop     | Similar espíritu; popular en RNNs                      |

No necesitas implementarlos a mano. En Keras/PyTorch:

#+begin_src python
# Keras
# model.compile(optimizer='adam', loss='mse')

# PyTorch
# optimizer = torch.optim.Adam(model.parameters(), lr=0.001)
#+end_src

*Para dummies*: Adam es un "descenso por gradiente inteligente" que suele funcionar bien
sin retocar tanto el learning rate. El concepto de fondo sigue siendo: seguir el
gradiente para bajar el error.

*** Resumen: las 7 ideas que debes llevarte

1. *Objetivo*: minimizar una función de error moviendo parámetros.
2. *Gradiente*: indica la dirección de mayor subida del error; vamos al contrario.
3. *Fórmula*: =parámetro_nuevo = parámetro_viejo − α · gradiente=.
4. *Learning rate α*: controla el tamaño del paso; ni muy grande ni muy pequeño.
5. *Batch / SGD / Mini-batch*: cuántos ejemplos usas por paso (mini-batch gana en la práctica).
6. *Mismo motor* para regresión lineal, logística y redes neuronales; cambia el error y los gradientes.
7. *En la vida real*: normaliza datos, prueba α, usa Adam, monitorea que el error baje.

**** Flujo mental cuando entrenas cualquier modelo

#+begin_src text
   Datos → Modelo con parámetros θ
              ↓
         Predicción ŷ
              ↓
         Error L(y, ŷ)
              ↓
         Calcular ∇L (gradiente)
              ↓
         θ ← θ − α·∇L
              ↓
         (repetir hasta que L sea aceptable)
#+end_src

Eso es descenso por gradiente. Todo lo demás son detalles, trucos y variantes sobre
esta misma idea.

**** Glosario rápido

- *Gradiente (∇)*: vector de derivadas; dirección de máximo crecimiento del error.
- *Descenso por gradiente (GD)*: algoritmo que resta α·gradiente para minimizar error.
- *Learning rate (α, lr)*: tamaño del paso en cada actualización.
- *Época (epoch)*: una pasada completa por todo el dataset (o por todos los mini-batches).
- *Iteración / paso*: una actualización de parámetros (un mini-batch en entrenamiento típico).
- *Batch size*: cuántos ejemplos por iteración en mini-batch.
- *SGD*: stochastic gradient descent; gradiente con un (o pocos) ejemplos.
- *Convergencia*: cuando los parámetros dejan de cambiar mucho y el error se estabiliza.
- *Divergencia*: cuando los parámetros explotan (NaN, oscilaciones enormes) por α mal elegido.




* Regresión Lineal Simple

** La idea

Es el modelo más sencillo del mundo: *dibujar la mejor línea recta* que pase entre
tus datos, para poder predecir.

Ejemplo: tienes las horas que estudiaron unos alumnos y su calificación. Quieres
predecir la calificación de alguien que estudió 5 horas. Trazas una recta y listo.

#+begin_src text
   calificación
      100 |                         •
          |                    •  ╱
       75 |               •  ╱
          |          •  ╱   ← la "mejor recta"
       50 |     •  ╱
          |  •╱
       25 |╱
          └────────────────────────────── horas de estudio
             1   2   3   4   5   6
#+end_src

** La fórmula (la ecuación de la recta de siempre)

#+begin_src text
   ŷ = m·x + b
#+end_src

- =ŷ= → lo que predecimos (ej. la calificación).
- =x= → el dato de entrada (ej. horas de estudio).
- =m= → la *pendiente*: cuánto sube ŷ por cada unidad de x.
- =b= → la *ordenada al origen* (intercepto): el valor de ŷ cuando x = 0.

("Simple" significa una sola variable de entrada x. Si hubiera varias, sería
"regresión lineal múltiple": =ŷ = m₁·x₁ + m₂·x₂ + ... + b=.)

** ¿Cómo se encuentra la "mejor" recta?

"Mejor" = la que tiene el *MSE más bajo* (¿recuerdas la Parte 1?). Es decir, la
recta cuya distancia (al cuadrado) a todos los puntos es mínima. Por eso este
método clásico se llama *Mínimos Cuadrados* (Ordinary Least Squares).

#+begin_src text
   Buscamos m y b que minimicen:

           1     n
   MSE =  ───  · Σ  (yᵢ − (m·xᵢ + b))²
           n    i=1
#+end_src

Se puede resolver de dos formas:
1. Con una *fórmula exacta* (mínimos cuadrados) — hay ecuación cerrada.
2. Con *gradiente descendente* — igual que la red neuronal (¡es el mismo truco!).

** Regresión lineal en Python (con scikit-learn)

#+begin_src python
import numpy as np
from sklearn.linear_model import LinearRegression

# Horas estudiadas (X) y calificación (y)
X = np.array([[1],[2],[3],[4],[5],[6]])   # sklearn quiere X en 2D
y = np.array([20, 40, 55, 70, 80, 95])

import numpy as np
x = np.array([1,2,3,4,5,6])
y = np.array([20,40,55,70,80,95])

x_media, y_media = x.mean(), y.mean()
# pendiente = covarianza(x,y) / varianza(x)
m = np.sum((x - x_media)*(y - y_media)) / np.sum((x - x_media)**2)
b = y_media - m * x_media

* Regresión Logística

** ¡Ojo! No sirve para predecir números, sino para CLASIFICAR

A pesar de llamarse "regresión", la regresión logística se usa para responder
preguntas de *sí/no* (clasificación binaria):

- ¿Este correo es spam? (sí / no)
- ¿El paciente tiene la enfermedad? (sí / no)
- ¿El alumno aprobará? (sí / no)

Devuelve una *probabilidad* entre 0 y 1. Si es > 0.5 → "sí", si es < 0.5 → "no".

** El problema con una recta y la solución: la sigmoide

Si usáramos una recta, podríamos obtener predicciones absurdas como 1.7 o −0.3, y
las probabilidades tienen que estar entre 0 y 1. Solución: pasar el resultado por la
*función sigmoide*, que aplasta cualquier número al rango (0, 1).

#+begin_src text
   probabilidad = sigmoide(z),   donde   z = m·x + b

               1
   sigmoide(z) = ─────────
               1 + e^(−z)

   Forma de "S":

    1 |               ________
      |            ╱
   0.5|          ╱   ← en z=0 vale 0.5
      |       ╱
    0 |______╱
      └──────────────────────  z
         −    0    +
#+end_src

- Si z es muy grande y positivo → sigmoide ≈ 1 (seguro que "sí").
- Si z es muy negativo → sigmoide ≈ 0 (seguro que "no").
- Si z = 0 → sigmoide = 0.5 (totalmente indeciso).

** ¿Cómo aprende? (otro tipo de error)

Aquí NO se usa el MSE, sino un error llamado *entropía cruzada* (log loss), que
castiga fuertemente cuando el modelo está *seguro y equivocado*. Pero la idea de
fondo es idéntica: ajustar m y b con gradiente descendente para minimizar el error.

#+begin_src text
   Log loss (una muestra):
     si real=1 →  −log(p)        (te castiga si p es bajo)
     si real=0 →  −log(1 − p)    (te castiga si p es alto)
#+end_src

** Regresión logística en Python

* Árboles de Decisión (Decision Trees)

** La idea: un juego de "20 preguntas"

Un árbol de decisión clasifica (o predice) haciendo una serie de *preguntas de
sí/no* encadenadas, como cuando juegas a adivinar un animal:

#+begin_src text
                    ¿Edad > 30?
                    /         \
                  sí           no
                  /             \
        ¿Ingreso > 50k?      ¿Estudiante?
          /       \            /      \
        sí        no         sí        no
        │          │          │         │
     APRUEBA   RECHAZA     APRUEBA   RECHAZA
     crédito   crédito     beca      beca
#+end_src

- Cada *nodo* es una pregunta sobre una característica (edad, ingreso...).
- Cada *rama* es una respuesta (sí/no, o un rango).
- Cada *hoja* (nodo final) es la decisión/predicción.

Es el modelo más *fácil de interpretar*: puedes leer el árbol y entender por qué
tomó cada decisión. Por eso se dice que es una "caja blanca" (no una caja negra).

** ¿Cómo decide qué preguntar primero?

El árbol elige la pregunta que *mejor separa* los datos, es decir, la que deja los
grupos resultantes lo más "puros" posible (que en cada grupo casi todos sean de la
misma clase). Para medir la "pureza" usa métricas como:

- *Impureza de Gini*: probabilidad de clasificar mal si eligieras al azar. 0 = puro.
- *Entropía / Ganancia de información*: cuánta "incertidumbre" elimina la pregunta.

#+begin_src text
   Antes de preguntar:   🔴🔴🔴🔵🔵🔵   (mezclado, impuro)

   Pregunta buena → separa así:
        rama sí:  🔴🔴🔴   (puro)      rama no: 🔵🔵🔵   (puro)

   Pregunta mala → separa así:
        rama sí:  🔴🔴🔵   (aún mezclado)  rama no: 🔴🔵🔵
#+end_src

El árbol prueba todas las preguntas posibles y elige la que más aumenta la pureza.
Luego repite el proceso en cada rama (esto se llama partición *recursiva*).

** El peligro: sobreajuste (overfitting)

Si dejas crecer el árbol sin límite, acaba memorizando los datos de entrenamiento
(una hoja por cada dato) y falla con datos nuevos. Es como un alumno que memoriza
las respuestas del examen en vez de entender la materia.

Soluciones (hiperparámetros):
- =max_depth= → limitar la profundidad (cuántas preguntas encadenadas).
- =min_samples_leaf= → exigir un mínimo de datos por hoja.
- *Poda* (pruning) → recortar ramas poco útiles.

** Árbol de decisión en Python

árboles en un *Random Forest* o *Gradient Boosting* (varios árboles votan juntos).

** Glosario rápido

- *Modelo*: la "fórmula" que aprende de los datos para predecir.
- *Peso / parámetro*: los números internos que el modelo ajusta al entrenar.
- *Entrenar / aprender*: buscar los parámetros que minimizan el error.
- *Error / Loss / Función de costo*: número que mide qué tan mal predice el modelo.
- *MSE*: error cuadrático medio (para números / regresión).
- *Log loss / entropía cruzada*: error típico para clasificación.
- *Gradiente*: la "pendiente" del error; indica hacia dónde crece el error.
- *Gradiente descendente*: mover los pesos en contra del gradiente para bajar el error.
- *Learning rate*: tamaño del paso al ajustar los pesos.
- *Época (epoch)*: una pasada completa por todos los datos de entrenamiento.
- *Backpropagation*: método para calcular los gradientes de todos los pesos de la red.
- *Sigmoide*: función en forma de S que convierte cualquier número en (0,1).
- *Overfitting*: el modelo memoriza en vez de generalizar; falla con datos nuevos.



* Árboles de decisión 

 Los árboles de decisión (DT) son un método de aprendizaje supervisado
 no paramétrico que se utiliza para la clasificación y la regresión. El
 objetivo es crear un modelo que prediga el valor de una variable de
 destino mediante el aprendizaje de reglas de decisión simples
 deducidas de las características de los datos. Un árbol puede verse
 como una aproximación constante por partes.

 en el siguiente ejemplo, los árboles de decisión aprenden de los datos
 para aproximarse a una curva sinusoidal con un conjunto de reglas de
 decisión if-then-else. Cuanto más profundo es el árbol, más complejas
 son las reglas de decisión y más ajustado es el modelo

 #+ATTR_ORG: :width 500
 [[file:img/dtr.png]]


** Elementos

 Los árboles de decisión están formados por nodos, vectores de números,
 flechas y etiquetas.

 - Cada nodo se puede definir como el momento en el que se ha de tomar
   una decisión de entre varias posibles, lo que va haciendo que a
   medida que aumenta el número de nodos aumente el número de posibles
   finales a los que puede llegar el individuo. Esto hace que un árbol
   con muchos nodos sea complicado de dibujar a mano y de analizar
   debido a la existencia de numerosos caminos que se pueden seguir.
 - Los vectores de números serían la solución final a la que se llega
   en función de las diversas posibilidades que se tienen, dan las
   utilidades en esa solución.
 - Las flechas son las uniones entre un nodo y otro y representan cada
   acción distinta.
 - Las etiquetas se encuentran en cada nodo y cada flecha y dan nombre
   a cada acción.

** Algunas ventajas de los árboles de decisión son:

 - Fácil de entender y de interpretar. Los árboles se pueden
   visualizar.

 - Requiere poca preparación de datos. Otras técnicas a menudo
   requieren la normalización de datos, es necesario crear variables
   ficticias y eliminar valores en blanco. Sin embargo, tenga en cuenta
   que este módulo no admite valores faltantes.

 - El costo de usar el árbol (es decir, predecir datos) es logarítmico
   en la cantidad de puntos de datos usados ​​para entrenar el árbol.

- Capaz de manejar datos numéricos y categóricos. Otras técnicas
  suelen estar especializadas en analizar conjuntos de datos que
  tienen un solo tipo de variable.

- Capaz de manejar problemas de múltiples salidas.

- Utiliza un modelo de caja blanca. Si una situación dada es
  observable en un modelo, la explicación de la condición se explica
  fácilmente mediante lógica booleana. Por el contrario, en un modelo
  de caja negra (por ejemplo, en una red neuronal artificial), los
  resultados pueden ser más difíciles de interpretar.

- Posibilidad de validar un modelo mediante pruebas estadísticas. Eso
  permite dar cuenta de la fiabilidad del modelo.

- Tiene un buen desempeño incluso si sus supuestos son algo violados
  por el verdadero modelo a partir del cual se generaron los dato

** Desventajas de los árboles de decisión:

- El aprendizaje de los  árboles de decisión pueden crear árboles demasiado
  complejos que no generalizan bien los datos. Esto se llama
  sobreajuste. Para evitar este problema, son necesarios mecanismos
  como la poda, establecer el número mínimo de muestras requeridas en
  un nudo de la hoja o establecer la profundidad máxima del árbol.

- Los árboles de decisión pueden ser inestables porque pequeñas
  variaciones en los datos pueden generar un árbol completamente
  diferente. Este problema se mitiga mediante el uso de árboles de
  decisión dentro de un conjunto.

- Las predicciones de los árboles de decisión no son uniformes ni
  continuas, sino aproximaciones constantes por partes, como se ve en
  la figura anterior. Por lo tanto, no son buenos para la
  extrapolación.

- Se sabe que el problema de aprender un árbol de decisión óptimo es
  NP-completo bajo varios aspectos de optimización e incluso para
  conceptos simples. En consecuencia, los algoritmos prácticos de
  aprendizaje del árbol de decisiones se basan en algoritmos
  heurísticos, como el algoritmo voraz, en el que se toman decisiones
  localmente óptimas en cada nodo. Dichos algoritmos no pueden
  garantizar la devolución del árbol de decisión globalmente
  óptimo. Esto se puede mitigar entrenando varios árboles en un alumno
  de conjunto, donde las características y las muestras se muestrean
  aleatoriamente con reemplazo.

- Hay conceptos que son difíciles de aprender porque los árboles de
  decisión no los expresan fácilmente, como XOR, paridad o problemas
  de multiplexor.

- Los aprendices de árboles de decisión crean árboles sesgados si
  dominan algunas clases. Por lo tanto, se recomienda equilibrar el
  conjunto de datos antes de ajustarlo al árbol de decisión.

** Índice de Gini en Árboles de Decisión

*** ¿Qué es el índice de Gini?
El **índice de Gini** es una medida de impureza o diversidad que se
utiliza en algoritmos de aprendizaje automático, como los Árboles de
Decisión. Se utiliza para medir qué tan puras o mezcladas están las
clases en un nodo del árbol.

*** Concepto del Índice de Gini
El índice de Gini mide la probabilidad de que una muestra elegida al
azar sea clasificada incorrectamente si se la asigna una clase de
manera aleatoria basada en la distribución de clases en ese nodo.

- **Un nodo puro** tendrá un índice de Gini igual a 0, lo que significa
  que todas las muestras en ese nodo pertenecen a la misma clase.
- **Un nodo impuro** tendrá un índice de Gini mayor, lo que indica que
  hay una mezcla de clases en ese nodo.

*** Fórmula del Índice de Gini
El índice de Gini se calcula como:

\[
Gini = 1 - \sum_{i=1}^{n} p_i^2
\]

Donde:
- **n** es el número de clases.
- **p_i** es la proporción de ejemplos de la clase **i** en el nodo.

*** Ejemplos del Índice de Gini

**** Ejemplo 1: Nodo completamente puro
Si todas las muestras en un nodo pertenecen a una sola clase, entonces el índice de Gini es **0**.

- Ejemplo: Si hay 100 muestras y todas pertenecen a la clase "A", entonces \( p_A = 1 \) y las demás probabilidades son 0.
- Cálculo:
#+BEGIN_SRC python
Gini = 1 - (1^2) = 0
#+END_SRC
Esto indica que el nodo es completamente puro.

**** Ejemplo 2: Nodo con clases mixtas
Si las muestras están distribuidas equitativamente entre varias clases, el índice de Gini será mayor.

- Ejemplo: Si hay 100 muestras, con 50 de la clase "A" y 50 de la clase "B", entonces \( p_A = 0.5 \) y \( p_B = 0.5 \).
- Cálculo:
#+BEGIN_SRC python
Gini = 1 - (0.5^2 + 0.5^2) = 1 - 0.25 - 0.25 = 0.5
#+END_SRC
Esto indica que el nodo contiene una mezcla de clases.

**** Ejemplo 3: Nodo con clases desbalanceadas
Si en un nodo hay 90 muestras de la clase "A" y 10 de la clase "B", el índice de Gini será más bajo que en el caso de clases equilibradas.

- Ejemplo: \( p_A = 0.9 \) y \( p_B = 0.1 \).
- Cálculo:
#+BEGIN_SRC python
Gini = 1 - (0.9^2 + 0.1^2) = 1 - 0.81 - 0.01 = 0.18
#+END_SRC
Esto indica que la mayoría de las muestras pertenecen a una clase, pero el nodo aún no es completamente puro.

*** Interpretación del Índice de Gini
- **Gini = 0**: El nodo es completamente puro (todas las muestras pertenecen a una sola clase).
- **Gini cercano a 0**: La mayoría de las muestras pertenecen a una clase, pero hay algo de mezcla.
- **Gini cercano a 1**: El nodo es muy impuro, con una mezcla casi uniforme de varias clases.

*** Uso en Árboles de Decisión
El árbol de decisión selecciona la característica que minimiza el índice de Gini en los nodos hijos, creando divisiones que generen nodos lo más puros posibles.

*** Ejemplo del Índice de Gini en tu Árbol de Decisión
En el nodo raíz del árbol de decisión que generaste, el **índice de Gini** es 0.667:
- Esto indica que hay una mezcla significativa de clases en ese nodo.
- El árbol de decisión divide los datos para reducir el índice de Gini en los nodos hijos.

* Reconocimiento de rostros con OpenCV — Eigenfaces, Fisherfaces y LBPH

El *reconocimiento de personas* por vision artificial responde a: *¿quién
es esta cara?* No basta con *detectar* un rostro (localizar rectángulo);
hace falta *identificarlo* contra un conjunto de identidades conocidas.

Flujo típico con OpenCV:

#+begin_example
  Dataset (carpetas/persona)
           |
           v
  [carga + etiquetas] --> [entrenamiento FaceRecognizer] --> modelo.xml
                                                              |
  Webcam --> Haar Cascade --> ROI 100x100 --> predict() <-----+
                                              |
                                              v
                                    etiqueta + confidence
#+end_example

** Requisitos prácticos

- Python + OpenCV con módulo =cv2.face= (paquete =opencv-contrib-python=).
- Clasificador Haar: =haarcascade_frontalface_alt.xml= (ruta relativa
  =data/= o la de tu instalación).
- Dataset organizado así:

#+begin_example
  /home/likcos/pruebacaras/
  ├── lalo/
  │   ├── 001.jpg
  │   ├── 002.jpg
  │   └── ...
  ├── ana/
  │   └── ...
  └── juan/
      └── ...
#+end_example

Cada subcarpeta =nombre/= es una *clase* (una persona). Las imágenes de
entrenamiento deben ser *recortes de rostro* en escala de grises y,
idealmente, del mismo tamaño (p. ej. 100×100).

** Convención del =predict=

=faceRecognizer.predict(roi)= devuelve =(etiqueta, confidence)=:

- *etiqueta*: índice de la persona (0, 1, 2, … según el orden de entrenamiento).
- *confidence*: medida de distancia / disimilitud. *Cuanto más baja, mejor* el match.

Por eso en los tres métodos usamos =if confidence < umbral= para aceptar
identidad, y en caso contrario =Desconocido=.

#+begin_quote
Los umbrales (=2800=, =500=, =70=) son *heurísticos*: dependen del dataset,
del tamaño del ROI y de la iluminación. Ajústalos empiricamente.
#+end_quote

** Eigenfaces

*** Idea

Un *Eigenface* (cara propia) es un vector propio de la matriz de
covarianza del espacio de imágenes de rostros. Sirovich y Kirby
propusieron el enfoque; Turk y Pentland lo usaron para clasificación.

Informalmente: cada cara se ve como un punto en \(\mathbb{R}^{W \cdot H}\) (un píxel = una coordenada). PCA encuentra direcciones de máxima varianza; las primeras *eigenfaces* forman una base compacta. Cualquier rostro se aproxima como:

\[
\text{cara} \approx \text{cara promedio}
  + \alpha_1 e_1 + \alpha_2 e_2 + \cdots + \alpha_k e_k
\]

donde \(e_i\) son eigenfaces y \(\alpha_i\) los coeficientes. Guardar solo esos coeficientes ocupa mucho menos que la imagen completa.

*** La matriz de covarianza

Antes de “vectores propios” y PCA, conviene entender *qué tabla* estamos diagonalizando.

**** Analogía de un minuto

Imagina que mides de mucha gente solo *dos* cosas: *altura* y *peso*.

- Algunas personas altas pesan más, otras no.
- Si cuando sube la altura *también tiende a subir* el peso, esas dos medidas *covarían* (se mueven juntas).
- Si una sube y la otra baja, covarian *en sentido contrario*.
- Si una no dice nada de la otra, la covarianza es ~0 (casi independientes).

La *matriz de covarianza* es una tablita que resume *todas* esas relaciones a la vez:

#+begin_example
              altura          peso
altura    var(altura)     cov(altura,peso)
peso   cov(peso,altura)    var(peso)
#+end_example

- En la *diagonal*: varianza de cada variable (“¿cuánto se dispersa sola?”).
- Fuera de la diagonal: covarianza entre pares (“¿cómo se mueven juntas?”).
- Es *simétrica*: =cov(A,B) = cov(B,A)=.

**** En una cara: cada píxel es una “variable”

Una imagen 100×100 en escala de grises son *10 000 números* (un brillo por píxel).  
Si tienes *N* fotos de rostros, tienes *N* puntos en un espacio de 10 000 dimensiones.

La matriz de covarianza de las caras responde, a grosso modo:

#+begin_quote
Cuando el píxel de la ceja izquierda se pone más claro en muchas fotos,
¿el píxel de la ceja derecha suele hacer lo mismo? ¿Y el de la punta de la nariz?
#+end_quote

Es decir: *qué zonas de la cara suelen variar juntas* a lo largo del dataset.

Por eso:

1. Restamos la *cara promedio* (centramos los datos: “solo nos importa la diferencia respecto al típico”).
2. Calculamos la covarianza entre píxeles (o un truco equivalente, *snapshot PCA*, cuando hay *muchos* píxeles y *pocas* fotos).
3. Los *vectores propios* de esa matriz son direcciones de máxima varianza conjunta → las *eigenfaces*.
4. El *valor propio* asociado dice “cuánta variación captura” esa dirección.

**** Mini-fórmula

Para datos centrados \(X\) (filas = caras, columnas = píxeles), una forma clásica es:

\[
C = \frac{1}{N-1} X^{\top} X
\]

- \(C_{ii}\): varianza del píxel \(i\).
- \(C_{ij}\): covarianza entre píxeles \(i\) y \(j\).

No tienes que calcularla a mano: OpenCV lo hace dentro de =EigenFaceRecognizer.train()=. Lo importante es la intuición:

#+begin_example
  muchas caras  -->  “¿qué patrones de variación se repiten?”
                         |
                         v
              matriz de covarianza
                         |
                         v
              vectores propios = eigenfaces
                         |
                         v
           “ingredientes” para recomponer caras
#+end_example

**** Ejemplo numérico 

Supón solo *2 píxeles* y *3 caras* ya centradas (restada la media):

| Cara | píxel A | píxel B |
|------+---------+---------|
| 1    | +2      | +2      |
| 2    | +0      | +0      |
| 3    | −2      | −2      |

A y B *siempre suben o bajan juntos* → covarianza alta y positiva.  
La primera “eigenface” apuntaría hacia algo como “(A y B juntos)”: el patrón dominante del dataset.

Si en cambio A sube cuando B baja, la covarianza sería negativa y el patrón dominante sería otro.

**** Por qué importa para Eigenfaces

Sin la matriz de covarianza, PCA no sabría *hacia dónde* “estirar” el espacio de caras.  
Con ella, el algoritmo encuentra esas pocas direcciones (=eigenfaces=) que explican la mayor parte de la variación entre rostros… y luego representa cada persona con pocos números \(\alpha_i\) en vez de 10 000 píxeles.

*** Generación (PCA)

1. Apilar imágenes de entrenamiento (mismo tamaño, grises).
2. Calcular la cara media y la matriz de covarianza (o el truco de snapshot PCA cuando hay más píxeles que muestras).
3. Obtener vectores propios; quedarse con los \(k\) de mayor valor propio.
4. Proyectar cada cara al subespacio y clasificar por distancia (p. ej. Euclidean) al vecino más cercano.

Las eigenfaces se ven como patrones claros/oscuros: simetría, vello facial, línea del cabello, tamaño de nariz/boca… Otras parecen “casi caras” abstractas.

*** Más allá del rostro

La misma idea (*eigenimages*) aparece en escritura, lectura de labios, voz, lenguaje de señas e imágenes médicas.

*** Ventajas y limitaciones

| Aspecto                 | Comentario                                                                                   |
|-------------------------+----------------------------------------------------------------------------------------------|
| Fortaleza               | Simple, base histórica, reducción de dimensión clara                                         |
| Debilidad               | Muy sensible a *iluminación* y alineación; PCA maximiza varianza, no separación entre personas |
| Umbral típico (ejemplo) | =confidence < 2800= (ajustar)                                                                  |

** Fisherfaces

*** Idea

*Fisherfaces* extiende el enfoque de PCA con *Análisis Discriminante Lineal* (LDA / Fisher). No solo comprime la varianza: busca proyecciones que *maximicen la separación entre clases* (personas) y *minimicen la dispersión dentro de cada clase*.

Por eso suele resistir mejor cambios de iluminación y expresión cuando hay varias muestras por persona.

*** Pipeline

1. *Preproceso*: tamaño, orientación e iluminación lo más uniformes posible.
2. *PCA*: reduce dimensionalidad (y hace viable LDA cuando hay pocos ejemplos frente a muchos píxeles).
3. *LDA*: encuentra ejes que separan clases.
4. *Clasificación*: proyectar y asignar por distancia / k-NN en el espacio discriminante.

#+begin_quote
PCA maximiza *varianza global*. LDA maximiza *razón entre-clases / dentro-de-clase*. Fisherfaces combina ambos.
#+end_quote

*** Ventajas y limitaciones

| Aspecto | Comentario |
|---------+------------|
| Fortaleza | Mejor discriminación entre personas; más robusto a iluminación/expresión que Eigenfaces puro |
| Debilidad | Necesita *varias imágenes por persona*; con una sola muestra por clase degradada a PCA-like |
| Umbral típico (ejemplo) | =confidence < 500= |

** LBPH (Local Binary Patterns Histograms)

*** Idea

A diferencia de Eigenfaces/Fisherfaces (que miran el rostro *global* como vector), *LBPH* describe *textura local*: patrones de intensidad alrededor de cada píxel.

Pasos:

1. *Dividir* el rostro en celdas (rejilla).
2. *LBP* por píxel: comparar el centro con sus 8 vecinos → código binario de 8 bits (0–255).
3. *Histograma* de códigos LBP en cada celda.
4. *Concatenar* histogramas → vector de características de la cara.
5. *Reconocer* por distancia (p. ej. chi-cuadrado / Euclidiana) al histograma más cercano de la base.

*** Por qué es popular

- Simple y rápido.
- Aguanta mejor variaciones de *iluminación* (LBP es relativamente invariante a cambios monotónicos de brillo).
- Adecuado para tiempo real y sistemas embebidos.
- No exige un preproceso tan estricto como PCA/LDA globales.

*** Ventajas y limitaciones

| Aspecto | Comentario |
|---------+------------|
| Fortaleza | Iluminación, velocidad, actualización incremental posible en OpenCV |
| Debilidad | Menos “geométrico”; depende de alineación local y resolución de la rejilla |
| Umbral típico (ejemplo) | =confidence < 70= |
